# 根因分析

## 直接原因

- stall / stop 路径把“停 worker”和“释放 gate”混在了一起。
- 释放 gate 时，系统没有足够可靠的证据证明旧 owner 已经安全结束。
- summary 聚合没有强制同一 generation 边界，导致状态混拼。

## 深层原因

- 运行正确性过度依赖本地内存态。
- 缺少统一的代际边界。
- 观测层和执行层之间没有足够清晰的收口协议。

## 为什么会被放过

- 表面上看，worker 还能继续跑，UI 也能展示部分状态，问题不容易第一眼看出来。
- 直到出现重复派发、session/thread/turn 混拼、或 stale 现场时，风险才会显性化。

## 事故级结论

这不是“加一点日志”能解决的问题，必须把：

- 释放顺序
- generation fencing
- 协作式 interrupt
- summary 过滤

一起作为合同修正。
