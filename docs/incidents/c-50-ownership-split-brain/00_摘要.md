# 事故摘要

## 发生了什么

`C-50` 暴露了一个所有权分裂问题：当旧 owner 还没有被明确确认安全结束时，系统就过早释放了 dispatch gate，导致同一 `issue_id` 可能再次被派发，产生并发多跑 worker、跨代状态混拼、以及“旧线程还没死，新线程又被拉起”的假现场。

## 为什么这是事故

这个问题影响的是运行态正确性，不是单纯的代码风格或局部命名问题。它会直接破坏“同一 issue 任一时刻只有一个当前主人”的基本合同，也会让后续排查者无法只靠一条运行记录判断真实状态。

## 影响范围

- 同一 `issue_id` 的 ownership 判定
- stall / stop / retry 的释放顺序
- `RunTrace` / `RunStateStore` / snapshot 的跨代字段混拼
- dashboard 上的线程、session、turn 状态可读性

## 这次的修复边界

本事故的修复思路已经进入 `docs/changes/c-50-ownership-gate/`，核心方向是：

- 先请求协作式 stop
- 再确认当前 turn 已收口
- 最后才决定是否释放 gate
- summary 只能消费同一 generation 的事件
