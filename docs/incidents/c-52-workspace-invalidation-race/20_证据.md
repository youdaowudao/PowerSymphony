# 证据

## 1. active owner 真相源存在，但它只解决消息控制面

已证实的代码事实：

- `elixir/lib/symphony_elixir/orchestrator.ex` 里的 `SymphonyElixir.Orchestrator.State` 定义了 `running`、`claimed`、`blocked_claims`、`retry_attempts`。
- dispatch gate 直接依赖这些字段决定是否还能起新 worker。
- `handle_spawned_issue_transition_result/3` 在 dispatch 成功后写入 `running`，并把 issue 放入 `claimed`。
- `block_issue_claim/3` 会在不安全释放 ownership 时保留 claim，并写入 `blocked_claims`。

这说明：

- 当前 active owner 真相源是真实存在的；
- 但它是 orchestrator 的**消息控制面真相源**，不是 workspace 资源本身的代际身份。

## 2. generation / interrupt / blocked-claim 机制已经存在

已证实的代码事实：

- `orchestrator.ex:new_run_instance_id/0` 已为每次 dispatch 生成 `run_instance_id`。
- `handle_spawned_issue_transition_result/3` 已把 `run_instance_id` 落入 `running_entry`。
- `agent_runner.ex` 已把 `run_instance_id` 贯穿到：
  - `worker_runtime_info`
  - `codex_update`
  - `run_result`
- `event_normalizer.ex` 已把 `run_instance_id` 标准化到 raw event。
- `run_state_store.ex:events_for_generation/2` 已按 `run_instance_id` 过滤事件。
- `codex/app_server.ex` 的 `receive_loop/8` 已支持 `{:interrupt_codex_turn, run_instance_id, reason}`，并发送 `turn/interrupt`。
- `orchestrator.ex:request_stalled_issue_interrupt/4` 已在 stall 命中时触发 cooperative interrupt，并启动 `{:stop_grace_timeout, issue_id, run_instance_id}`。
- `orchestrator.ex:block_unconfirmed_stall_stop/3` 已在 stop 未确认时转 `blocked_claim(reason: :remote_stop_unconfirmed)`。

这说明：

- 当前主线已经有一套面向 stalled run 的生命周期保护；
- 但这套 protection 主要保护“哪些消息属于当前代”，没有保护“当前 workspace 资源属于哪一代”。

## 3. workspace 路径没有代际身份，只按 issue identifier 固定复用

已证实的代码事实：

- `Workspace.create_for_issue/2` 最终通过 `workspace_path_for_issue(safe_id, worker_host)` 计算路径。
- `safe_identifier/1` 只基于 issue identifier 做规范化，不带 `run_instance_id`。
- 本地 `ensure_workspace/2` 在目录已存在时直接返回 `{:ok, workspace, false}`。
- 远端 `ensure_workspace/2` 也在目录已存在时直接复用，并返回 `created=0`。
- retry 路径会沿用已有 `workspace_path`。

这说明：

- workspace 当前是按 issue 固定复用的共享资源；
- 它没有自己的 generation 身份；
- old run、active run、retry、startup sweep、cleanup side-path 都可能继续碰同一个物理路径。

## 4. terminal cleanup 路径旁路了现有收口协议

已证实的代码事实：

- `reconcile_issue_state/4` 在 issue 进入 terminal state 时，直接调用 `terminate_running_issue(state, issue.id, true)`。
- `terminate_running_issue/3` 当前顺序是：
  1. `cleanup_issue_workspace`
  2. `terminate_task`
  3. 删除 `running / claimed / blocked_claims / retry_attempts`

与 stalled path 对比：

- stalled path 已经是“请求 cooperative interrupt -> 等待 grace -> 决定 retry 或 blocked claim”。
- terminal path 却没有先走这套协议，而是直接 cleanup workspace。

这说明：

- terminal cleanup 的危险窗口是真实存在的；
- 但它应该被放在“共享 workspace 没有 generation fencing”的更大背景里理解。

## 5. cleanup side-path 同样是粗粒度删目录

已证实的代码事实：

- `run_terminal_workspace_cleanup/0` 启动时会扫描 terminal issues，并按 `identifier` 直接 cleanup workspace。
- `handle_terminal_retry_issue/4` 在 issue 已 terminal 时直接 cleanup workspace。
- `release_terminal_blocked_claim_issue/2` 在 issue 已 terminal 时同样直接 cleanup workspace。

这说明：

- `terminate_running_issue/3` 不是唯一危险点；
- 多条 terminal lifecycle side-path 都共享同一个“按 identifier 粗粒度删目录”的模型。

## 6. 当前 cleanup 只按 identifier 定位目录，不带 run-instance 防护

已证实的代码事实：

- `cleanup_issue_workspace/2` 最终调用 `Workspace.remove_issue_workspaces(identifier, worker_host)`。
- `remove_issue_workspaces/2` 只按 `safe_identifier(identifier)` 计算目录路径。
- 当前 cleanup 并不校验“这个目录是否仍属于触发 cleanup 的旧 run_instance_id”。

这说明：

- 如果后续修复引入延迟 cleanup，却仍按 `identifier` 直接删目录，就必须正面处理“旧 cleanup 误删新 owner 复用 workspace”的风险。

## 7. `AppServer.stop_session/1` 当前没有可靠终态确认链

已证实的代码事实：

- `AppServer.stop_session/1` 当前只调用 `stop_port(port)`。
- `stop_port/1` 只尝试 `Port.close(port)`，没有可靠的“底层 terminal/app-server 已完全退出”的确认协议。

这说明：

- 即使把 cleanup 顺序调过来，如果删除前仍拿不到可靠终态证据，资源生命周期 race 仍然可能存在。

## 8. 外部会话没有统一的 per-turn validity gate

已证实的代码事实：

- `codex/app_server.ex:start_session/2` 只在 session 启动时执行一次 `validate_workspace_cwd/2`。
- `agent_runner.ex:run_codex_turns/6` 只在整条 run 开始时 `AppServer.start_session(...)` 一次。
- 后续 `do_run_codex_turns/5` 递归复用同一个 session 持续执行 `run_turn`。
- `continue_with_issue?/3` 只刷新 issue 状态，并决定继续还是结束。

当前没有统一校验去阻止：

- owner 已切换但当前会话仍继续跑；
- workspace 已被 invalidated 但 turn 仍继续尝试；
- issue 已经由其他 run 收口但旧会话仍继续工作。

## 9. cleanup 后没有留下明确 invalidation 语义

已证实的代码事实：

- `workspace.ex:remove/2` 本地路径是：
  1. `maybe_run_before_remove_hook`
  2. `File.rm_rf(workspace)`
- 远端路径也是 `before_remove` 后直接 `rm -rf`。
- 当前没有看到任何 tombstone、marker 文件、owner metadata 快照或 invalidation 记录写入。

因此 cleanup 发生后，workspace 只剩“物理上消失”这一种事实。

## 10. 为什么用户最终只看到 `cwd missing`

已证实的代码事实：

- `AppServer.start_port/2` 以 workspace 作为真实工作目录启动本地执行器。
- `thread/start` 与 `turn/start` 也继续把同一个 workspace 作为 `cwd` 传给 Codex。
- 当 cleanup 删掉该目录后，失败点就直接落在 `CreateProcess` / `cd` / `No such file or directory` 层。

换言之：

- 生命周期错误没有在语义层被接住；
- 它直接从文件系统错误冒到了用户面前。

## 11. 线程捕捉结果支持“像双开”，但不足以直接证明 duplicate thread

已证实的材料事实：

- 当前材料里只抓到一个已坐实的 orchestrator 顶层主会话：
  `019e29a2-51c7-7813-866d-e75c1094bf83`
- `Boyle` 以及其他派生线程都带 `parent_thread_id=019e29a2-...`，属于 subagent。
- 在已核对的 `symphony.log.2` 中，只看到一次 `C-52` 顶层 `Starting agent run` / `Starting worker attempt`。

但当前仍然不能写成“系统层面绝无第二顶层线程”，原因是：

- active owner 真相源仍然主要在 orchestrator 内存态；
- 当前没有全局可读的 owner ledger；
- 本次材料范围也不等于所有 host / 所有 orchestrator 进程。

## 12. 最小复现能证明什么

最小复现只需要证明一件事：

- 当执行器继续使用一个已经被删掉的 workspace 作为 `cwd` 时，系统当前只会给出底层路径错误，而不会自动给出 invalidation 语义。

这与 `C-52` cleanup 后顶层会话实际看到的错误形式一致。

因此它能支撑的结论是：

- 当前错误语义层级过低。

它不能单独证明：

- 一定存在第二个顶层线程；
- 或一定不存在第二个顶层线程。
