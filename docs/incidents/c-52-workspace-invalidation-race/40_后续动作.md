# 后续动作

本文件只记录 incident 视角下的后续动作，不在这里展开具体实现。
具体修复设计、任务拆分、验证矩阵进入对应的 `docs/changes/<change-id>/`。

## incident 已经收敛出的修复主问题

后续 change 文档必须先回答下面三个根问题，再谈 marker、gate、报错翻译：

1. 一个 workspace 在资源层到底如何证明“当前仍属于这个 run”
2. cleanup 之前如何证明“旧 run 已经真的停干净”
3. startup sweep / retry / blocked-claim cleanup 为什么有权在不看 live resource ownership 的情况下删目录

## change 方案必须显式规避的三个坑

1. 不能再造第二套 owner 真相源。
   当前 `claimed / running / blocked_claims / run_instance_id` 已经是现有控制面的真相源；change 应复用它们，而不是另起一套独立 owner registry。

2. invalidation record 不能被当成主修复。
   它最多只能是：
   - 可见性工具
   - 错误翻译依据
   - 调试证据载体

   它不能代替：
   - workspace generation 身份
   - cleanup fencing
   - run 终态确认

3. 必须覆盖所有 terminal cleanup side-path，而不只是 `terminate_running_issue/3`。
   至少要覆盖：
   - startup terminal workspace sweep
   - running issue terminal cleanup
   - retry issue terminal cleanup
   - blocked claim terminal cleanup
   - local 与 remote worker 路径

## 当前 incident 不打算直接回答的内容

下面这些内容应进入 change，而不是继续塞回 incident：

- workspace 资源身份具体如何编码
- run 终态确认链具体由谁提供
- 是否引入延迟删除、保留目录还是重命名隔离
- invalidation record 的最终落点和格式
- 测试矩阵与回归门禁

## 仍需补强的证据

如果后续要继续追“是否真的存在第二个顶层 worker”，还需要额外证据源：

- 更完整的 orchestrator / host 侧启动记录
- 可稳定去重的 active-run ledger
- 能跨 session、跨 host 对齐 owner 的统一标识

这部分当前仍是待排查分支，不应反客为主盖过本次已坐实的 workspace lifecycle fencing 事故。
