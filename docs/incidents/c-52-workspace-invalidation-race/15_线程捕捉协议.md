# 线程捕捉协议

## 目标

这份文档只回答一件事：

- 怎样尽量完整地抓出 `C-52` 相关线程
- 怎样绕开 turn 噪音
- 当前证据为什么还不能 100% 排除“另一个顶层线程”

## 去噪原则

这次排查里，不能把 `turn` 当线程。

- 一个真实线程，只按一个 `session_meta.id` 计数一次。
- `Continuation guidance`、`turn/start`、`turn completed` 都只是同一线程里的多轮 turn，不应被重复计成新线程。
- 线程谱系必须以 session 文件首行 `session_meta` 为准，而不是以中间消息、turn 数或 commentary 数量为准。

## 推荐抓取顺序

1. 先按 workspace 抓所有 session 首行。
   规则：
   - `cwd == /home/ss/data/projects/symphony-workspaces/C-52`
2. 再按 `parent_thread_id == 019e29a2-51c7-7813-866d-e75c1094bf83` 抓派生线程。
3. 最后再用 `symphony.log*` 里的 `Starting agent run` / `Starting worker attempt` 对照顶层启动记录。

这样做的原因是：

- session 首行能区分“顶层线程 / 子代理线程 / 外部线程”
- log 能证明“系统何时声明启动一次顶层 run”
- 两者交叉后，才能避免把 turn 或子代理误写成 duplicate thread

## 判定字段

### 顶层线程

满足以下特征：

- `session_meta.payload.id` 是一个新的 thread id
- `thread_source` 不是 `subagent`
- `originator` 通常是 `symphony-orchestrator` 或外部入口
- 没有 `parent_thread_id`

### 子代理线程

满足以下特征：

- `thread_source == subagent`
- `source.subagent.thread_spawn.parent_thread_id` 指向父线程
- 可能有 `agent_nickname` / `agent_role`

### turn 噪音

下列内容不能单独算线程：

- `Continuation guidance`
- `turn/start`
- `Completed agent turn`
- `turn_count`
- `premature_turn_end`

这些都只说明同一线程继续跑下一轮 turn。

## 这套协议现在能证明什么

- 能证明 `Boyle` 不是第二个顶层 worker。
- 能证明大量“看上去像又起了一个线程”的记录，其实只是子代理或 turn 延续。
- 能证明当前已核对到的 `C-52` workspace session 集合里，只有一个已坐实的 orchestrator 顶层线程。

## 这套协议现在还证明不了什么

它还不能 100% 证明“绝无第二个顶层线程”，原因有三：

- active owner 的真相源不在 workspace，也不在 session 文件，而在 orchestrator 进程内存态。
- `Starting agent run` 目前只在已检查到的 `symphony.log.2` 中出现一次；这不足以代表所有 host / 所有 orchestrator 进程。
- 外部会话如果不复用同一个 `cwd`，或者日志不落在当前这组文件里，单靠这组材料无法自动把它补全出来。

所以当前更严谨的写法只能是：

- 已抓到的一组线程里，没有第二个已坐实的顶层线程
- 但当前日志体系还不足以把“系统层面不可能存在第二个顶层线程”写成铁结论

## 为什么用户仍会主观感受到“像双开”

- 同一个 issue 下确实会出现多条 session 文件。
- turn 继续、子代理派生、PR 已 merged、workspace 被 cleanup，这几件事在时间上交叠。
- 当旧会话继续执行时，系统不给“你已失效”的语义，只给 `cwd missing`。

于是用户从表面上看到的是：

- 我手里的线程还在说话
- 另一个人已经推了 PR 并 merge
- 我的 cwd 又突然没了

这非常容易被感知成“同一张 issue 被双开了两个线程”。
