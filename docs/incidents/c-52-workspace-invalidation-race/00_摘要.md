# 事故摘要

## 一句话结论

`C-52` 当前最强、最稳的结论不是“已经证实 duplicate thread”，而是：

- 主线已经给**消息状态**做了 generation 保护；
- 但 workspace 这个**物理资源**没有 generation 身份，只按 issue identifier 固定复用；
- cleanup、retry、startup sweep、旧会话续跑都可能继续碰同一个物理路径；
- terminal cleanup 还会先删目录，再停 worker；
- session / terminal 退出又没有可靠终态确认链；
- 所以问题最终表现成“像双开线程”“像目录被别人突然清掉”“像系统失控”。

## 已证实的事实

### 1. active owner 真相源存在，但它只解决消息控制面

- 当前 active owner 真相源主要是 `SymphonyElixir.Orchestrator.State` 里的 `running`、`claimed`、`blocked_claims`。
- `run_instance_id` 已经用于消息代际过滤、trace 归并和 stalled run 的 stop 保护。
- 这套机制能回答“哪些消息属于当前 run”，但不能回答“当前这个 workspace 资源到底属于哪个 run”。

### 2. 当前主线已经有一套 generation / interrupt / blocked-claim 机制

- 每次 dispatch 已经生成 `run_instance_id`。
- `RunStateStore` 已经按 `run_instance_id` 做 generation 过滤。
- `AppServer.receive_loop/8` 已经支持 `{:interrupt_codex_turn, run_instance_id, reason}`，并发送 `turn/interrupt`。
- stalled run 命中后，orchestrator 已经会请求 cooperative interrupt，并在 stop 未确认时落到 `blocked_claim(reason: :remote_stop_unconfirmed)`。

这说明：

- 现在的问题不能简单写成“系统完全没有 lifecycle / ownership 保护机制”。
- 更准确的是：这套保护主要覆盖**消息状态**，没有覆盖 **workspace 资源生命周期**。

### 3. workspace 当前是按 issue 固定复用的共享资源

- workspace 路径由 `safe_identifier(identifier)` 决定，不带 `run_instance_id`。
- 目录已存在时，本地与 remote 路径都会直接复用。
- retry 路径也会沿用已有 `workspace_path`。

这意味着：

- old run、active run、retry、cleanup side-path 都可能继续碰同一个物理目录；
- 但系统没有 generation-aware 的资源 fencing 去证明“这个目录现在到底归谁”。

### 4. terminal cleanup 确实存在“先删目录，再停 worker”的硬伤

- issue 进入 terminal state 时，`reconcile_issue_state/4` 会直接走 `terminate_running_issue(..., true)`。
- `terminate_running_issue/3` 当前顺序是：
  1. `cleanup_issue_workspace`
  2. `terminate_task`
  3. 删除 `running / claimed / blocked_claims / retry_attempts`

这说明 terminal cleanup 的危险窗口是真实存在的。

### 5. cleanup side-path 同样按 identifier 粗粒度删目录

- startup terminal sweep 会按 terminal issue 的 `identifier` 直接 cleanup workspace。
- retry terminal cleanup 会直接 cleanup workspace。
- blocked-claim terminal cleanup 也会直接 cleanup workspace。

所以真正的问题不是某一条函数顺序错了，而是多条 lifecycle 路径共享同一个“粗粒度删目录”模型。

### 6. 删除前当前拿不到可靠 run 终态确认链

- stalled path 虽然已经有 cooperative interrupt，但 `AppServer.stop_session/1` 现在仍只是 `Port.close`。
- 当前没有一条统一的“底层 terminal/app-server 已可靠退出”的确认链，可作为 cleanup 的硬前提。

### 7. 外部会话没有统一的 per-turn validity gate

- `AppServer.start_session/2` 只在 session 启动时做一次 `validate_workspace_cwd/2`。
- `AgentRunner` 后续在同一 session 上递归执行多轮 `run_turn`。
- `continue_with_issue?/3` 只判断 issue 状态还应不应该继续，不判断“这条会话还有没有资格继续”。

### 8. 用户最终只能看到底层 `cwd missing`

- `AppServer` 用 workspace 作为真实 `cwd`。
- `thread/start` 和 `turn/start` 都继续复用这个路径。
- 目录被删之后，失败点直接退化到 `CreateProcess` / `cd` 层。

所以用户看到的是：

- `No such file or directory`
- `cwd missing`

而不是：

- `workspace_invalidated`
- `owner_changed`
- `run_closed`

## 当前不能下的结论

### 不能下的结论 1

不能把这次事故简单回退成旧的 `C-50 / duplicate-thread gate` 结论。

原因：

- 当前代码已经有 `run_instance_id`、generation 过滤、cooperative interrupt、blocked claim。
- 本次坐实的问题是：这些保护主要覆盖消息状态，没有覆盖 workspace 资源生命周期。

### 不能下的结论 2

也不能写成“已经证明绝无第二个顶层线程”。

原因：

- 在线程捕捉到的材料里，没有抓到第二个已坐实的顶层 `Starting agent run`。
- 但当前 owner 真相源不在全局可读存储里，而在单 orchestrator 进程内存态。
- 只凭这批 session / log，还不足以把“系统层面绝无第二顶层线程”写成铁结论。

## 为什么用户会强烈感知成“像双开”

因为用户看到的是下面这组组合拳：

1. 自己手里的线程还在继续执行
2. 另一个 run 已经推进了 PR、merge、Done
3. workspace 又在此期间被 cleanup 掉
4. 系统不给“你已失效”的上层语义，只给 `cwd missing`

在这种观感下，把问题理解成“同 issue 双开了两个线程”是非常自然的，不是错觉。
