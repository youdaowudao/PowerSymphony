# 根因分析

## 根结论

`C-52` 当前最强的根因结论是：

- 系统已经给**消息状态**做了 generation 保护；
- 但没有给 **workspace 资源生命周期** 做 generation fencing；
- workspace 按 issue identifier 固定复用，cleanup 与复用都不看代际；
- terminal cleanup 还会先删目录，再停 worker；
- session / terminal 退出也没有可靠终态确认链；
- 因此 old run、active run、cleanup side-path 都可能继续碰同一个物理路径，最后才以 `cwd missing` 暴露出来。

这不是一个单点 bug，而是一条资源生命周期合同断裂链：

1. workspace 资源没有 generation 身份
2. cleanup / retry / startup sweep 都按 identifier 粗粒度碰目录
3. terminal cleanup 旁路现有 stop 合同
4. 删除前拿不到可靠 run 终态确认
5. 外部会话续跑前也不会证明自己仍然有效
6. 最终把资源生命周期错误退化成文件系统错误

## 根因 1：workspace 是共享资源，但没有 generation fencing

当前系统已经用 `run_instance_id` 保护消息状态，却没有让 workspace 本身带上代际身份。

这带来三个直接后果：

- 同 issue 的 workspace 会被固定复用；
- cleanup 也按同一个 `identifier` 继续定位这个目录；
- old run、active run、retry、startup sweep 之间没有资源层的 fencing。

所以更深的主因不是“owner 看不见”，而是“资源没有代际身份”。

## 根因 2：terminal cleanup 路径绕开了现有的 cooperative stop 合同

这次代码核查后，最重要的新结论是：

- stalled run 已经走 cooperative interrupt 和 blocked-claim；
- terminal issue cleanup 却没有复用这套路径。

也就是说，当前系统内部其实已经存在一套比较保守的“先请求收口、无法确认就保留 claim”的思路；
但它保护的是消息生命周期，没有真正保护到 workspace 资源生命周期。

因此 `C-52` 的关键不是“完全缺一套 stop/invalidation 协议”，而是：

- 已有协议保护的是消息状态；
- terminal 分支在资源层旁路了它。

## 根因 3：cleanup side-path 同样按 identifier 粗粒度删目录

当前不只 running issue terminal cleanup 有问题。

startup terminal sweep、retry terminal cleanup、blocked-claim terminal cleanup 都有同样的问题：

- 看 issue terminal 就删目录；
- 不问当前目录是否仍属于触发 cleanup 的旧 run。

所以这不是单条代码顺序错误，而是多条资源生命周期路径共享同一缺口。

## 根因 4：删除前没有可靠终态确认链

即使把 terminal cleanup 顺序从“先删目录”改成“先请求 stop”，问题也不会自动消失。

因为当前 `AppServer.stop_session/1` 只做 `Port.close`，没有可靠的“底层 terminal/app-server 已真正退出”的确认链。

所以 cleanup 真正缺的不是单个 marker，而是“在删除资源之前，如何证明旧 run 已经停干净”。

## 根因 5：会话在每轮执行前不会证明自己仍然有效

当前 worker / 外部会话的续跑判断更像是：

- issue 还 active 吗
- turn 数还没跑满吗

而不是：

- 我是不是当前 owner
- 我的 `run_instance_id` 还有效吗
- 我现在要碰的这个 workspace 资源是否仍属于我

这使得旧会话即使已经不再合法，也会继续尝试拿旧 cwd 往下跑，直到撞到更底层的错误。

## 根因 6：workspace cleanup 没有语义化 invalidation 产物

当前 cleanup 的产物只有一件事：

- 目录被删掉了

但它没有留下：

- invalidation reason
- owner / run_instance_id 快照
- invalidated_at
- cleanup_source

因此 cleanup 对旧会话来说不是“一个可读状态变更”，而只是“脚下路径突然没了”。

这里的关键不是“marker 完全没用”，而是：

- 如果不先解决资源 generation fencing 和终态确认，
- 单独新增 invalidation record 很容易变成第二套半真相源，
- 最终只是把现在的 `cwd missing` 升级成更隐蔽的“假 invalidation”或“旧 cleanup 删掉新 run 目录”。

## 根因 7：错误翻译层缺失，导致用户只能看到 `cwd missing`

一旦目录不存在，当前执行路径直接在 `Port.open(..., cd: workspace)` 或统一执行器 `CreateProcess` 层面失败。

如果上层没有在这之前先把会话挡住，或在这之后把错误翻译成：

- `workspace_invalidated`
- `owner_changed`
- `run_closed`

那用户看到的就只会是：

- `No such file or directory`
- `cwd missing`

这正是当前事故的最终表象。

## 当前不能过度外推的部分

### 1. 不能写成“已经证实 duplicate thread”

因为当前材料还不足以坐实第二个非 subagent 顶层 worker。

### 2. 也不能写成“只是旧会话误以为自己还活着”

因为现在已经坐实：

- 活跃线程在 cleanup 前仍在正常推进；
- cleanup 发生时系统确实可能把 workspace 从活跃 run 脚下抽走。

所以这不是单纯的“旧会话主观错判”，而是系统真的给了它一个危险且不透明的失效路径。

## 根因排序

按当前证据强度排序：

1. workspace 资源没有 generation fencing
2. cleanup side-path 多且按 `identifier` 粗粒度删目录
3. terminal cleanup 旁路现有收口协议
4. 删除前没有可靠终态确认链
5. `per-turn validity gate` 缺失
6. `workspace invalidation` 语义缺失
7. `duplicate-thread / cross-process ownership split` 仍是待排分支，但不是本次已坐实主因
