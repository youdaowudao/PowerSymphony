# 单次变更模板

建议至少从一个入口文件开始，例如：

```text
docs/changes/<change-id>/
  README.md
  10_design.md
  20_plan.md
  90_verification.md
```

入口文件 `README.md` 或等价文件应至少包含：

- 目标
- 非目标
- 需求快照
- 成功标准
- 固定约束
- 风险
- 文档索引
- 若本 change 已经决定后续要修改哪些规则文件，可写“已纳入实施范围，待对应规则 diff 落地”，不要提前写成“已生效”
- 若命中 `观察层合同风险`，在同一份冻结件里附窄版 `contract matrix`

说明：

- 这里的“需求快照”不是复制整条 Linear 历史，而是给 repo 内 review 提供最小、稳定、可审查的目标说明。
- reviewer 应能只看 change 文档、diff 和测试，就判断实现是否命中需求。
- change 入口还应点名这次必须看的 `SPEC/` 或 `阶段规划/` 文档，避免 coder 自己猜。
- 若执行面存在活状态板、流程指标、`Change Review`、`Push Readiness` 等实时字段，统一留在 issue body / `## Codex Workpad`，不要在 change 文档里重复抄实时值。
- `contract matrix` 不是独立文档模板，而是 `frozen artifact` 的窄版附录。最小字段建议为：
  - `field / view`
  - `source of truth`
  - `allowed transform`
  - `must not infer`
- 若变更定义了 `contract checker`、`baseline lock` 或 `closeout` 主路径，必须同时写清失效条件与最小回退规则，避免只留下理想顺序。
