# 验证分层标准

## 用途

这份文档描述：当你写一个新仓库、一个高风险变更，或者一份长期 SPEC 时，应该把验证写到什么粒度，避免把“该写什么验证”漏掉。

## 核心原则

1. 验证要和风险等级匹配。
2. 文档要告诉后续执行者先验证什么、再验证什么。
3. 不要把所有事情都上升成 `make all`。
4. 不要把没有必要的验证写进小改动。
5. 高风险路径必须能说清楚证据。
6. 中途风险验证应条件触发，不是每张卡默认重门禁。
7. `Push Readiness` 只回答下一次 push 是否可做以及最小还缺什么，不替代 gate。

## 验证分层

### 1. 文档级验证

适用：

- 纯文档更新
- 目录重组
- 命名调整

写法：

- 只检查链接、落点、命名和入口是否一致
- 不强迫跑代码测试

### 2. 定向验证

适用：

- 单个功能或单个文档包的更新
- 能明确界定影响面

写法：

- 列出最小验证命令
- 说明期望结果
- 只验证改动面对应的行为

### 3. 收口验证

适用：

- 准备开 PR
- 准备更新 PR
- 准备更新 Linear

写法：

- 写清楚需要过的门禁
- 说明哪些结果算通过
- 说明哪些结果算失败

### 4. 高风险验证

适用：

- 并发
- 生命周期
- 持久化
- 权限 / 安全
- 启动 / 停止 / 清理
- 外部副作用

写法：

- 明确需要哪些证据
- 明确哪些场景必须人工确认
- 明确哪些路径不能靠默认回归代替

### 5. 条件触发的中途风险验证

适用：

- BEAM / typed core path
- event normalization
- state machine / concurrency 边界
- reviewer 明确标记 type / integration risk
- 返工后仍跨多个风险区的累计 diff

写法：

- 写清触发条件是什么
- 写清触发后要补的最小证明是什么
- 写清是否需要把下一次 push gate 升级
- 不要把它写成新的 state、审批层或长期阻塞理由

## 质量门禁文档应该写什么

如果你在写一个质量门禁相关的 SPEC，至少要回答：

- 门禁想保护什么
- 它拦什么风险
- 它不负责什么
- 哪些验证属于门禁的一部分
- 哪些验证属于更上层的收口流程
- 失败时怎么判断是文档问题、实现问题还是环境问题

## 新仓库怎么用

新仓库可以直接复制这份规则，并按自己的技术栈补两项：

- 本仓库默认的验证命令
- 本仓库高风险路径的额外门禁

## 不要这样写

- “写适当的验证”
- “按情况补充测试”
- “请做必要检查”

这类说法没有办法指导后续 coder。

应该写成：

- 具体验证命令是什么
- 期望输出是什么
- 失败时看哪里
- 触发中途风险验证时，最小补证是什么、是否影响下一次 push
