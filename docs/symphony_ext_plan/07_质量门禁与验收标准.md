# 07. 质量门禁与验收标准

## 1. 总原则

这套二开最大风险不是“写不出来”，而是：

- 改坏官方单项目运行链路；
- 多项目状态串线；
- Web 默认加载过多后台信息导致卡顿；
- trace 记录太粗，仍然看不清 Codex 在干什么；
- workflow 生成物与源文件不一致。

质量门禁必须围绕这些风险设计。

---

## 2. Gate 0：证据门禁

每轮 AI 执行前必须确认：

- 已阅读相关源码文件。
- 已查看当前 `git diff --stat` 与目标文件清单，并先判断本轮验证级别。
- 已说明当前改动是否触碰核心执行链路。
- 已说明保留的原行为。
- 已说明新增 API 是否可能破坏旧 API。

验证级别必须先判定，再决定是否运行测试命令：

- `L0 - 不运行测试`：文档-only、只读勘测、Linear 整理、规则梳理、纯说明文字修改。
- `L1 - 定向验证`：局部代码改动、非核心配置改动、局部 UI/接口改动，只运行与改动直接相关的最小验证命令。
- `L2 - 最终确认`：核心代码改动、测试/构建配置改动、影响单项目启动链路或执行链路的改动、重大修复、高风险改动，或 CI 已失败且本地已完成定向排查修复后，准备再次推送前的最终确认。

关于 `make -C elixir all` 的硬规则：

- 禁止默认运行 `make -C elixir all`。
- 开发阶段只跑定向测试，不跑完整 `make -C elixir all`。
- 准备开 PR 或准备更新 PR 之前，必须先跑 closeout gate：`mix format --check-formatted`、`mix lint` 和改动面对应的定向测试；测试命令必须显式带 `SYMPHONY_TEST_MAX_CASES`，默认值统一为 `4`。
- 只有判定为 `L2 - 最终确认` 时，才允许运行 `make -C elixir all`，且它不是日常开发命令，也不是复现工具。
- 如果 CI 失败，顺序固定为：先看 CI 报错，再做本地定向检测，随后修复问题，最后在修复完成后跑一次本地 `make -C elixir all` 做最终确认。
- 文档-only、只读勘测、Linear 整理阶段不得运行 `make -C elixir all`。
- 若未先基于 `git diff` 明确写出验证级别与理由，不得运行 `make -C elixir all`。
- 重型测试和 `make -C elixir all` 必须带资源监控，至少覆盖内存、swap、CPU、子进程/端口/worker 增长以及系统卡顿风险。
- 一旦监控发现资源不足，必须立即停止当前重型测试、回收现场、把并发从 `4` 降到 `2`，仍不稳再降到 `1`；`1` 还不稳就停止并报告机器承载不了当前 full gate。
- 每次测试结束后都必须做现场回收，清掉残留 worker / fake worker / background server、端口占用、临时文件/目录/日志、测试注入的环境变量和配置覆写，以及其他本轮产生的垃圾状态。

AI 每轮输出必须包含：

```text
已检查源码：...
git diff 判断：...
验证级别：L0 / L1 / L2
本轮不修改：...
本轮新增/修改：...
兼容性影响：...
```

## 3. Gate 1：单项目兼容门禁

任何阶段都必须保持：

```bash
./bin/symphony ./WORKFLOW.md
```

仍然可用。

说明：

- 这里描述的是必须保持的兼容性目标，不等于每轮都要默认实跑该链路。
- 只有当 `git diff` 显示触碰单项目启动链路、核心执行链路、workflow loader、测试/构建配置，或进入最终确认场景时，才提升到 `L2 - 最终确认`。

验收：

- 原单项目 CLI 不因控制面变更失效。
- 原 `/api/v1/state` shape 不被破坏。
- 原 `/api/v1/<issue_identifier>` 不被破坏。
- 原 `--logs-root`、`--port` 行为不被破坏。

---

## 4. Gate 2：多项目隔离门禁

必须证明：

- 项目 A 的 workflow 不会被项目 B 读取。
- 项目 A 的 logs root 不会写到项目 B。
- 项目 A 的 workspace root 不会写到项目 B。
- 项目 A worker crash 不影响项目 B worker。
- 项目 A worker port 不与项目 B 冲突。

验收样例：

```text
启动 A、B 两个项目。
停止 A。
B 仍然 running。
A 状态变 stopped。
控制面仍 running。
```

---

## 5. Gate 3：Web 轻量展示门禁

这是硬门禁。

项目总览页不允许加载：

- raw event；
- 完整 timeline；
- 完整 prompt；
- 完整 Linear payload；
- 完整 shell output；
- 完整 tool response。

验收方式：

- 检查 LiveView assigns。
- 检查 API 请求日志。
- 检查总览页 payload size。
- 检查打开总览页时没有调用 `/raw`、`/timeline`、`/events/:id`。

通过标准：

```text
/projects 只调用 summary API。
summary payload 不包含 raw_event / payload / prompt_body / shell_output 字段。
```

---

## 6. Gate 4：Codex 状态可读性门禁

每个 run summary 必须包含：

```text
current_phase
current_action
health
last_event_at
run_duration_seconds
session_id
thread_id
turn_id
turn_count
last_error
```

至少能区分：

```text
正在准备 workspace
正在启动 Codex
Codex 正在分析
Codex 正在读取文件
Codex 正在修改文件
Codex 正在运行 shell
正在等待 shell 返回
正在等待 linear_graphql 返回
最近一段时间没有新事件
疑似卡住
正在重试
需要人工处理
```

验收：

- fake Codex events 能驱动 phase/action/health 变化。
- quiet/stalled 时间阈值可测试。
- 状态短句不依赖前端推理。

---

## 7. Gate 5：懒加载门禁

详情层必须按需加载。

验收：

- Run 详情默认最多加载最近 50 条 timeline。
- Event payload 点击后才加载。
- Raw JSON 点击后才加载。
- 大 shell output 分页。
- prompt body 默认不加载。

通过标准：

```text
打开 Run 页面时没有一次性加载全量 raw events。
打开 Event 详情时只加载该 event 的 payload。
```

---

## 8. Gate 6：Trace 可靠性门禁

Trace 系统不能影响执行主链路。

要求：

- RawEventStore 写入失败时记录 warning，不让 run 失败。
- StateReducer 异常时 fallback 到 `unknown`，不让 orchestrator crash。
- Timeline 压缩失败时保留 raw event。
- payload 超大时截断或落文件，不塞内存。

验收：

- 模拟 trace 写入目录无权限。
- 模拟超大 payload。
- 模拟 malformed event。
- worker 不应因此退出。

---

## 9. Gate 7：Workflow 编译门禁

要求：

- generated 文件能被现有 workflow loader 读取。
- source hash 与 generated hash 对得上。
- 缺文件时不启动 worker。
- YAML 错误清晰展示。
- stale 状态显示在 Web。

验收：

```text
修改 workflow/prompt/20_execution_rules.md
控制面显示 workflow stale
执行 compile
生成新的 WORKFLOW.generated.md
控制面显示 current
worker 可启动
```

---

## 10. Gate 8：性能门禁

个人 WSL 环境第一版指标：

- 1-5 个项目同时配置。
- 2 个项目同时 running。
- 总览页刷新无明显卡顿。
- summary API p95 < 500ms。
- worker API timeout 可配置。
- 单个 run raw events 上千条时，总览页性能不受明显影响。

不要求：

- 企业级 100 项目并发。
- 复杂监控指标。
- 数据库索引优化。

---

## 11. Gate 9：安全与脱敏门禁

要求：

- 默认绑定 loopback。
- 不在 Web 总览显示 API key。
- Raw payload 也要脱敏常见 secret。
- worker process command 不在 UI 展示完整 env。
- 错误日志不要泄露 token。

验收：

- 人工 grep summary response，确认无 `LINEAR_API_KEY`。
- 模拟 payload 中带 Authorization header，raw 页面显示脱敏。

---

## 12. Gate 10：回滚门禁

每阶段都必须能回滚。

要求：

- 控制面可以关闭。
- worker 可以手动按单项目方式启动。
- 新增代码失败不破坏旧命令。
- 有清理 pid / port / logs 的说明。

---

## 13. AI 每轮交付格式

AI 每轮执行结束必须输出：

```text
1. 本轮目标
2. 修改文件
3. 保持不变的旧行为
4. 新增 API / 配置
5. 测试命令
6. 测试结果
7. 通过的质量门禁
8. 未完成/已知风险
9. 回滚方式
```

阶段 1 可以不提供测试结果，但必须明确说明本轮只完成仓库初始化、文档阅读、方向确认和控制面骨架。
从阶段 2 起，必须提供与本轮验证级别匹配的验证结果：

- `L0`：说明本轮为何无需运行测试，并明确禁止补跑 `make -C elixir all`。
- `L1`：提供定向验证命令、结果、覆盖范围与未覆盖边界。
- `L2`：提供最终确认命令、结果，以及为何需要提升到最终确认。

若本轮是文档-only、只读勘测或 Linear 整理，没有测试结果是允许的，但必须有清楚的 `git diff` 判定与免测理由。

---

## 14. Definition of Done

整个二开第一版完成的定义：

- 可配置多个项目。
- 控制面能启动/停止/重启每个项目 worker。
- Web 总览能显示项目状态和 counts。
- 项目详情能显示 issue/run 的 phase/action/health。
- Run 详情能显示压缩 timeline。
- Raw event/payload/shell/prompt 均按需加载。
- Workflow 源文件能生成 `WORKFLOW.generated.md`。
- 原单项目 CLI 仍可用。
- 质量门禁 0-10 全部通过。
