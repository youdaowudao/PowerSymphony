# Symphony 二次开发计划书：执行入口

版本：v1.0  
日期：2026-05-03  
适用对象：个人 WSL 环境，基于 OpenAI Symphony Elixir 原型机进行二次开发。  
主线：只做官方 Elixir 原型机二开，不做换语言重写。

---

## 0. 一句话定盘

把 OpenAI Symphony Elixir 原型机二开成一个**多项目 Codex 执行控制台**：

- 每个项目仍然运行一套单项目 Symphony worker。
- 新增一个控制面统一管理多个项目 worker。
- Web 前端只默认展示轻量摘要；详细事件、原始 payload、长日志必须点击后按需加载。
- `WORKFLOW.md` 不取消，改成人维护模块化源文件，运行时使用生成后的 `WORKFLOW.generated.md`。
- 后续所有 AI 执行都沿这个方向，不再讨论换语言。

## 2. 最重要的二次复核调整

我在复核官方源码后，把多项目实现方式从“同一个 BEAM 进程内启动多个 Orchestrator”收紧为：

> **控制面管理多个单项目 Symphony OS worker。**

原因：当前官方源码里 `Workflow`、`Config`、`WorkflowStore` 明显以单全局工作流为中心：CLI 只接收一个 workflow path，`Workflow` 从 `Application.get_env(:symphony_elixir, :workflow_file_path)` 取全局路径，`WorkflowStore` 以模块名注册成单例。直接改成 in-process 多项目，会先撞上全局状态去耦合，风险比预期大。

因此第一版多项目应采用更稳的形态：

```text
Control Plane
  ├─ 管理 symphony.projects.yaml
  ├─ 为每个项目生成/校验 WORKFLOW.generated.md
  ├─ 为每个项目分配 logs_root 与内部 API port
  ├─ 启动/停止/重启每个单项目 Symphony worker 进程
  └─ Web 聚合各 worker 的轻量 summary

Project Worker A
  └─ 原单项目 Symphony，读取项目 A 的 WORKFLOW.generated.md

Project Worker B
  └─ 原单项目 Symphony，读取项目 B 的 WORKFLOW.generated.md
```

这个方案更符合“别把多项目能力塞进一个核心 Orchestrator”的原则，也更利于避免项目之间状态串线。

## 3. 交给 AI 执行时的总提示词

可以把下面这一段作为每轮 AI 执行的固定前缀：

```text
你正在基于 OpenAI Symphony Elixir 原型机做二次开发。禁止换语言，禁止全量重写，禁止把多个项目塞进一个核心 Orchestrator。主线是：新增多项目 Control Plane，管理多个单项目 Symphony worker；每个 worker 仍读取自己的 WORKFLOW.generated.md、logs_root、workspace_root、内部 API port。Web 总览页只能读取轻量 summary，不允许默认拉取 raw event、完整 timeline、完整 prompt、完整 shell output。Codex 线程观测必须通过后端 state reducer 生成 phase/action/health，再给前端展示。每轮修改必须先按当前阶段要求执行：阶段 1 先完成创建仓库、阅读文档、确认方向和骨架搭建；从阶段 2 起必须先读当前源码和相关测试，并按《07_质量门禁与验收标准.md》先基于 `git diff` 判断验证级别，再给出设计差异、修改文件、测试命令、回滚方式。禁止默认运行 `make -C elixir all`；只有核心代码改动、测试/构建配置改动、或 PR 前最终重核验时才允许。文档-only、只读勘测、Linear 整理阶段不得运行该命令。不要生成小型任务列表，按本文档的阶段和质量门禁推进。
```

---

## 4. 不允许偏离的硬约束

1. **不换语言。** 只做 Elixir 原型机二开。
2. **不重写核心执行链路。** Orchestrator / AgentRunner / Codex AppServer 的主行为先保留。
3. **多项目先用 OS worker 隔离。** 不在第一版做 in-process 多 Orchestrator。
4. **前端默认只看摘要。** Raw event / payload / prompt / shell output 全部懒加载。
5. **状态由后端推导。** 前端不从原始 Codex 事件自行推理。
6. **每项目独立 workflow。** 控制面只管项目列表，不接管所有 prompt/rules/hooks。
7. **WORKFLOW 是生成物，不是消失。** 人维护模块化源文件，worker 读取 generated 文件。
8. **质量门禁先于功能堆叠。** 没有 summary API、trace 分层、回归测试，不继续扩 Web 细节。

---

## 5. 最小可交付目标

第一版不追求复杂产品化，只交付一个可干活的本地控制台：

- 能从 `symphony.projects.yaml` 读取多个项目。
- 能启动/停止/重启每个项目的单项目 worker。
- 能在 Web 上看到所有项目的轻量状态。
- 能进入某个项目，看运行中的 issues/runs。
- 能进入某个 run，看 Codex 线程当前 phase/action/health。
- 能按需打开 timeline、event detail、raw payload。
- 能用模块化 workflow 源文件生成 `WORKFLOW.generated.md`。
- 能通过质量门禁验证“没有把后台重信息默认推到前端”。
- 第一阶段至少能让控制面管理 2 个项目，并保持各项目 `workflow`、`workspace`、日志和运行状态独立。

# 01. 技术路线决策与二次复核

## 1. 最终技术路线

主线固定为：

> **基于 OpenAI Symphony Elixir 原型机做二次开发，新增多项目控制面、Web 观测增强、Codex 线程状态可视化、Workflow 模块化生成。**

不做：

- 不做换语言重写。
- 不做一个巨型多项目 Orchestrator。
- 不做复杂产品级 Web SaaS。
- 不把后台全部事件默认推给前端。

---

## 2. 官方资料给出的边界

官方 `README.md` 把 Symphony 定位为低调工程预览，并给出两种使用方式：基于 spec 自己实现，或使用 Elixir experimental reference implementation。仓库页面还显示目前没有正式 release。  
来源：`https://github.com/openai/symphony`

官方 Elixir README 明确说明：Elixir 目录是当前基于 `SPEC.md` 的实现；它是 evaluation-only prototype；运行机制是轮询 Linear、创建 per-issue workspace、启动 Codex app-server、发送 workflow prompt，并保持 Codex 工作直到完成。  
来源：`https://github.com/openai/symphony/blob/main/elixir/README.md`

官方 `SPEC.md` 明确说 Symphony 要解决四个问题：把 issue 执行变成 daemon workflow、隔离 per-issue workspace、让 workflow policy 留在 repo 的 `WORKFLOW.md`、提供足够观测能力来调试多个并发 agent run。  
来源：`https://github.com/openai/symphony/blob/main/SPEC.md`

官方 `SPEC.md` 同时把 rich web UI 和 multi-tenant control plane 列为 non-goals。这并不代表不能二开，而是说明这些不是官方核心 conformance 的一部分，必须作为外层产品化扩展来做，不能让它们影响 orchestrator 正确性。  
来源：`https://github.com/openai/symphony/blob/main/SPEC.md`

---

## 3. 二次复核后的关键修正

### 3.1 原先方向

原先方向是：在一个 Elixir 应用内增加 `ProjectRegistry`、`ProjectSupervisor`、`ProjectWorker`，每个 ProjectWorker 内部运行一套 `WorkflowStore + Orchestrator + AgentRunner`。

### 3.2 复核后发现的重大问题

官方源码当前有明显单项目全局假设：

- CLI 只接受一个 `WORKFLOW.md` 路径，未设计多 workflow 配置。
- `Workflow.workflow_file_path()` 从 `Application.get_env(:symphony_elixir, :workflow_file_path)` 获取全局 workflow path。
- `WorkflowStore.start_link()` 用 `name: __MODULE__` 注册单例。
- `Config.settings()` 调用 `Workflow.current()`，也沿用了全局 workflow。
- 当前 Router / Presenter 的 API shape 也只围绕一个 orchestrator snapshot。

这意味着如果直接在同一个 BEAM 进程里跑多个 ProjectWorker，第一阶段就要去耦合 `Workflow`、`WorkflowStore`、`Config`、`Orchestrator` 的全局依赖。这不是不能做，但会把风险集中到最脆的执行主链路。

### 3.3 修正后的主线

第一版采用：

> **多项目 Control Plane 管理多个单项目 Symphony OS worker。**

每个项目 worker 是一个独立 OS 进程：

```bash
./bin/symphony \
  --logs-root /path/to/logs/project-a \
  --port 4101 \
  /path/to/project-a/WORKFLOW.generated.md
```

控制面只做：

- 读取 `symphony.projects.yaml`；
- 为每个项目分配 worker port / logs root / workflow path；
- 启动、停止、重启 worker；
- 周期性拉取 worker 的轻量 summary；
- Web 统一展示；
- 点进详情时按需代理 worker 的 timeline/event/raw API。

这个修正把多项目复杂度放在外层，而不是改坏核心执行链路。

---

## 4. 为什么这个方向正确

### 4.1 保持单项目 worker 的稳定性

官方原型机当前的强项是单项目运行链：Linear → workspace → Codex app-server → workflow prompt → retry/reconciliation。不要第一步就破坏这个链路。

### 4.2 更容易定位问题

每个项目 worker 独立进程，天然隔离：

- workflow path 不会串；
- logs root 不会串；
- Application env 不会串；
- WorkflowStore 单例不会冲突；
- 一个项目 crash 不直接污染另一个项目内存状态。

### 4.3 Web 性能边界更清楚

控制面只聚合 summary，不订阅所有 raw events。每个 worker 自己持有详细 trace；控制面只在用户点击某个 run 时向该 worker 拉取详情。

### 4.4 更符合官方 spec 的 extension 思路

`SPEC.md` 明确说 status surface / HTTP server 是 observability/control extension，并且 dashboard/API 不应成为 orchestrator correctness 的必要条件。控制面作为外层扩展，符合这个边界。

---

## 5. 重大风险与处理方式

### 风险 A：多个 OS worker 会消耗更多资源

处理：个人 WSL 环境通常是少量项目，先接受这个开销。第一版通过项目 enabled 开关、每项目 `agent.max_concurrent_agents`、worker 启停控制来压住资源。等项目数量明显变多，再评估 in-process 多 worker 去全局化。

### 风险 B：跨项目总并发不好统一控制

处理：第一版不做全局精确调度。每项目 workflow 内设置自己的 `agent.max_concurrent_agents`。控制面只做粗粒度项目级启停和状态告警。不要为了全局并发去重写核心 Orchestrator。

### 风险 C：worker 内部 trace 不够细

处理：先增强单项目 worker 的观测接口：`RunTrace`、`RunSummary`、`TimelineSummary`、`RawEventStore`。这些增强仍在单项目上下文里，风险远低于同时做 in-process 多项目。

### 风险 D：Web 页面变成日志大屏，卡顿

处理：硬性分层：summary 默认展示，timeline 点击加载，raw payload 再次点击加载。LiveView 只接收 summary diff，不接收 raw event。

### 风险 E：WORKFLOW.generated.md 与源文件不一致

处理：workflow compiler 必须输出生成时间、源文件 hash、manifest hash；worker 启动前校验 generated 文件是最新的；Web 页面显示 workflow stale 状态。

---

## 6. 最终决策表

| 领域 | 主选 | 不选 | 原因 |
|---|---|---|---|
| 运行底座 | 官方 Elixir 原型机 | 换语言重写 | 当前目标是二开完整机器，不是重建机器 |
| 多项目 | 控制面管理多个单项目 OS worker | 同进程多 Orchestrator | 源码存在全局 workflow/config/store，直接多实例风险大 |
| Web | Phoenix/LiveView 控制台增强 | 新建重前端 | 官方已有 Phoenix dashboard/API，先沿用 |
| 观测 | 后端 summary reducer + 懒加载 raw | 前端订阅 raw events | 避免卡顿和信息过载 |
| Workflow | 每项目模块化源文件 → generated workflow | 一个超级总 workflow | 项目策略必须跟项目仓库一起维护 |
| 数据存储 | 内存 summary + JSONL raw trace + logs | 一开始上数据库 | 官方 spec 不要求持久 DB，个人环境先轻量 |

---

## 7. 不再重新打开的争论

后续执行中不再讨论：

- 是否换语言；
- 是否用一个超级 orchestrator 承载所有项目；
- 是否把所有项目规则放进一个大 workflow；
- 是否把原始 Codex 事件实时推给前端；
- 是否第一版就做企业级权限/多用户/远程部署。

只有当第一版稳定运行、且出现明确瓶颈时，才重新评估这些问题。

# 02. Superpower 前置头脑风暴

## 1. 要解决的真实问题

这次二开不是为了“更多功能”，而是为了让 Symphony 成为一个能长期干活的本地执行控制台。

当前原型机的主要不足：

1. **单项目限制。** 官方原型机一次读一个 workflow，真实使用需要同时管理多个项目。
2. **Web 信息太粗。** 当前 dashboard/API 只显示少量运行信息，难以判断 Codex 线程到底在干什么。
3. **Codex 事件不可读。** 原始事件太细，当前展示又太粗，缺少中间层状态摘要。
4. **Workflow 过重。** `WORKFLOW.md` 既是配置又是长 prompt，又承载项目规则，人工维护压力大。
5. **后台信息不能乱推前端。** 如果总览页默认拉取所有 raw event / payload / logs，会造成资源浪费和卡顿。

---

## 2. 最终产品形态

目标产品不是复杂云平台，而是一个本地控制台：

```text
Symphony Control Plane
  ├─ 读取 symphony.projects.yaml
  ├─ 管理多个 Project Worker OS 进程
  ├─ 聚合轻量 summary
  ├─ 展示项目总览 / 项目详情 / Run 详情
  ├─ 按需代理 timeline / event / raw payload
  └─ 编译每项目 workflow 源文件
```

每个 Project Worker 仍然是单项目 Symphony：

```text
Project Worker
  ├─ WORKFLOW.generated.md
  ├─ Linear project_slug
  ├─ workspace_root
  ├─ logs_root
  ├─ Codex app-server
  └─ RunTrace / Summary API
```

---

## 3. 用户视角的核心体验

### 3.1 项目总览页

一打开 Web 页面就能看到：

```text
项目 | worker 状态 | running | quiet | stalled | retrying | needs_attention | 最近活动 | token | 最后错误
```

这个页面不能加载 raw event。

### 3.2 项目详情页

进入某个项目后，看到该项目所有活跃 issue/run：

```text
Issue | Linear 状态 | 当前阶段 | 当前动作 | 健康状态 | Codex thread | turn | 最近活动 | 持续时间
```

### 3.3 Run 详情页

进入某个 run 后，看压缩时间线：

```text
12:01:03 领取 Linear 事项
12:01:05 准备 workspace
12:01:08 启动 Codex session
12:01:14 Codex 正在分析需求
12:01:46 Codex 正在修改文件
12:02:10 正在运行测试命令
12:04:30 最近 2 分钟无新事件，标记 quiet
```

### 3.4 Event 详情页

点某条事件才看 payload 摘要和必要的原始字段。

### 3.5 Raw 调试页

只有调试时打开完整 JSON / shell output / prompt / Linear response。

---

## 4. 成功标准

### 4.1 可用性成功标准

- 能配置 2 个以上项目。
- 能从一个 Web 控制台启动/停止/重启项目 worker。
- 任意一个项目 worker 出错，不影响控制面和其他项目 worker。
- 总览页能一眼看出谁在工作、谁变慢、谁卡住、谁需要人工处理。

### 4.2 可观测性成功标准

每个 run 至少有：

- `current_phase`
- `current_action`
- `health`
- `last_event_at`
- `run_duration_seconds`
- `session_id`
- `thread_id`
- `turn_id`
- `turn_count`
- `last_meaningful_event`
- `last_error`
- `next_retry_at`

### 4.3 性能成功标准

- 项目总览页不加载 raw event。
- 项目详情页不加载完整 timeline。
- Run 详情页默认只加载最近 N 条压缩事件。
- Event payload 点击后懒加载。
- LiveView 推送只推 summary diff。
- 1-5 个项目同时运行时，Web 不明显卡顿。

### 4.4 工程成功标准

- 改动不破坏单项目原有启动方式。
- 原有 `./bin/symphony ./WORKFLOW.md` 仍可用。
- 新增 `./bin/symphony_control --config symphony.projects.yaml --port 4100`，作为本地开发/测试示例端口；`4000` 保留给本地常驻生产线程。
- 增加 fake worker / fake Codex / fake Linear 测试。
- 每个阶段可回滚。

---

## 5. 非目标

第一版明确不做：

- 企业多用户权限系统。
- 云端部署控制台。
- 拖拽式工作流编辑器。
- 复杂 DAG 工作流引擎。
- 跨项目精确全局调度器。
- 数据库型日志平台。
- 对核心 Codex app-server 协议做大改。
- 将所有项目揉入一个 `WORKFLOW.md`。

---

## 6. Superpower 头脑风暴结论

### 6.1 最有价值的能力

最有价值的不是“显示更多日志”，而是：

> **用少量状态字段表达 Codex 当前到底在干什么，以及它是否还在推进。**

因此 Web 设计要围绕三元组：

```text
current_phase + current_action + health
```

### 6.2 最危险的错误方向

最危险的是把后台事件当成前端状态：

```text
错误方向：Codex raw event -> LiveView -> 页面直接显示
```

正确方向：

```text
Codex raw event
  -> EventNormalizer
  -> StateReducer
  -> RunSummary
  -> LiveView summary diff
```

### 6.3 最稳的多项目边界

多项目不是把 core orchestrator 改成多项目，而是让每个项目继续拥有自己的单项目 worker。控制面只聚合、管理和代理。

### 6.4 Workflow 的正确边界

`WORKFLOW.md` 不能消失，因为它是项目策略契约。正确做法是让它成为生成物，而不是人工维护的巨文档。

---

## 7. 关键设计原则

1. **外层包裹，内核稳定。** 多项目能力放在控制面，不塞进核心 orchestrator。
2. **后台厚，前端薄。** 后台记录足够多，前端默认展示足够少。
3. **摘要优先，raw 懒加载。** 所有 raw payload 都要有明确点击动作才加载。
4. **状态可判断。** 不只显示 running，要显示正在做什么、多久没动、是否异常。
5. **项目独立。** workflow、workspace、logs、worker port 都按项目隔离。
6. **生成而非手写巨文档。** Workflow 人维护小文件，运行时吃 generated。
7. **质量门禁保护方向。** 每阶段必须证明没有把核心链路改坏，没有把前端做重。

## 2. 阶段 1：创建仓库、阅读项目文档、确认方向与控制面骨架

### 目标

完成项目初始化和第一版控制面骨架，为后续多项目扩展建立稳定入口。

### 主要产出

- 本地仓库创建完成，并已同步远端主线。
- 已阅读 `DOCS/` 中的项目规划文档，并确认主线方向。
- `symphony.projects.yaml` schema。
- 至少 2 个项目的配置样例。
- `ProjectConfig` 解析与校验。
- `ProjectRegistry` 内存状态。
- 新 CLI：`./bin/symphony_control --config ./symphony.projects.yaml --port 4100`，作为本地开发/测试示例端口；`4000` 保留给本地常驻生产线程。
- 控制面 Web/API 能显示项目列表、配置校验结果和基础配置态/占位态运行状态。

### 必须确认

- 控制面至少能管理 2 个项目。
- 每个项目有独立 `workflow_generated`、`workspace_root`、`logs_root` 和运行状态占位字段。
- 项目方向以本套规划文档为准，不偏离多项目 Control Plane 主线。
- 原单项目 CLI 不被破坏。
- 阶段 1 的状态仅限配置态、校验态和 `not_started` 一类占位态，不提前实现真实 worker 生命周期。

### 阶段门禁

- 配置错误能清晰展示。
- 项目 ID 校验防止路径穿越和重复。
- 控制面至少能管理 2 个项目。
- 各项目 `workflow`、`workspace`、日志和状态占位字段不串线。
- 不影响原单项目 CLI。
- 对存在未完成前置依赖的 issue，调度器不能仅因其处于 active state 就直接派发。
- 真实 worker 启停、健康检查、stdout/stderr 落盘统一放到阶段 2。

### M1-5：前置依赖识别与自动阻塞基线修正

#### 目标

修正当前单项目 Symphony worker 的调度基线：当 issue 仍存在未完成前置依赖时，即使该 issue 已被人工置为 `In Progress` 一类 active state，也不能被调度器直接派发执行。

#### 为什么放在 M1

- 这不是临时 bug 备注，而是后续多项目控制面必须依赖的调度语义基线。
- 如果不先修正，后续阶段一旦开始更依赖 Symphony 自主推进 issue，仍会退回到“人工维护下游长期停在 `Todo`”的脆弱流程。
- 阶段 2 会开始真正管理多个项目 worker；在那之前，必须先明确单项目 worker 的依赖阻塞语义。

#### 主要产出

- 当前调度行为与目标行为的文档化说明。
- 对 `blocked_by` 的统一调度规则：不只拦 `Todo`，而是拦所有可执行 active issue。
- 首次派发与 retry 派发使用同一套依赖阻塞判断。
- 针对依赖链顺序的最小验证用例。

#### 必须确认

- 不能依赖人工长期维持下游 issue 停在 `Todo`。
- 调度器必须自己判断前置依赖是否全部进入 terminal state。
- 依赖阻塞规则属于执行正确性，不属于 Web 展示层逻辑。
- 该修正不改变项目方向，不引入复杂 DAG 引擎。

#### 阶段门禁补充

- 上游 issue 未完成、下游 issue 已在 `In Progress` 时，下游仍不能被派发。
- 上游 issue 进入 terminal state 后，下游 issue 才重新具备可调度资格。
- retry 路径与首次派发路径对依赖阻塞的判断一致。

### M1-6：`turn/completed` 与错误续跑语义修正

#### 目标

修正单项目 Symphony worker 在 `turn/completed`、run 结束判定与 active issue continuation 上的基线语义，避免“turn 正常结束但 ticket 未闭环”被错误放大为连续续跑。

#### 为什么放在 M1

- 这不是单纯日志文案问题，而是单项目执行链的正确性问题。
- 如果不先修正，后续多项目控制面会建立在错误的 turn/run 边界上。
- 阶段 2 之前必须先冻结单项目 worker 的 continuation 语义，不然后续 worker 生命周期和调度策略会继续放大这类误续跑。

#### 主要产出

- turn 级完成与 run 级完成的语义拆层说明。
- active issue 下错误续跑边界的规划结论。
- `AgentRunner` 与 `Orchestrator` 的 run 级结果边界说明。
- `max_turns` 之后合法 continuation 路径的保留原则。
- 对应专题参考文档集合：`DOCS/plan_rerun_fix/`。

#### 必须确认

- `turn/completed` 仍然只是协议层的 turn 级终态，不冒充 ticket 完成。
- 主修点在 `agent_runner.ex` / `orchestrator.ex` 边界，不在 `app_server.ex` 协议接收本身。
- 外层 continuation 不能简单删除，因为它还承担 `max_turns` 打满后的合法续跑职责。
- 后续实现若要引入 run 级结构化终态，必须同时设计 `AgentRunner` 到 `Orchestrator` 的结果上报机制。

#### 阶段门禁补充

- `turn/completed` 不再被系统外观或编排逻辑误读为任务完成。
- 对 active issue，turn 级完成不能自动升级成健康的 run 完成。
- 错误收口必须与 timeout / crash / success 分开处理。
- 兼容性上继续保持原单项目 CLI 与 `./bin/symphony ./WORKFLOW.md` 不被破坏。

### M1-7：M1final 门禁收口与兼容性验收

#### 目标

对 M1 做阶段性收口，验证 M1 的计划边界、兼容性、配置读取、项目占位展示和最小 smoke check 都成立，为进入阶段 2 做清晰交接。

#### 为什么放在 M1 最后

- 这张卡不是补功能，而是防止 M1 在边界未对齐时“假完成”。
- 当前 M1 已不只包含最初的配置和控制面骨架，还补入了两项执行语义基线修正，因此需要单独的最终门禁项统一收口。
- 没有这一步，阶段 2 开始时很容易重新争论哪些边界已经冻结、哪些仍未完成。

#### 主要产出

- M1 已交付清单。
- M1 保持不变的旧行为清单。
- M1 已通过的门禁项清单。
- M1 明确未做项与进入阶段 2 的交接说明。
- 最小 smoke check 结果。

#### 必须确认

- M1 不要求完成真实 Linear / Codex / worker E2E。
- M1 必须保持原单项目 CLI 可用。
- 控制面至少能管理 2 个项目，并且只展示配置态、校验态和占位态。
- 真实 worker 启停、健康检查、stdout/stderr 落盘仍然统一放到阶段 2。

#### 阶段门禁补充

- 验证 `./bin/symphony ./WORKFLOW.md` 入口未被破坏。
- 验证 `symphony_control` 能读取样例配置并展示至少 2 个项目的占位态。
- 验证本阶段未提前引入真实 worker 生命周期能力。
- 输出进入阶段 2 前的明确 handoff。

## 3. 阶段 2：Project Worker 进程管理

### 目标

控制面能管理多个单项目 Symphony OS worker。

### 主要产出

- `ProjectProcessManager`。
- worker 启动命令生成。
- worker 健康检查。
- 启动/停止/重启 API。
- worker stdout/stderr 文件落盘。
- worker crash 后状态可见。

### worker 启动模式

每个项目启动为独立 OS 进程：

```bash
./bin/symphony \
  --logs-root <project.logs_root> \
  --port <project.worker_port> \
  <project.workflow_generated>
```

控制面只绑定主 Web 端口，例如 `4000`。worker 端口只绑定 loopback，例如 `127.0.0.1:4101`。
本地开发或测试若需要手动启动控制面，请改用非 `4000` 端口，例如 `4100` 或 `4123`，避免与本地常驻生产线程冲突。

### 关键设计

- 控制面保存 pid 文件。
- worker crash 不让控制面 crash。
- worker API 不直接暴露给外网。
- worker logs root 与 workspace root 均按项目隔离。
- 控制面不要解析 worker raw log 来判断主状态；优先调用 worker summary API。

### 阶段门禁

- 同时启动两个 dummy worker。
- 停止一个 worker 不影响另一个。
- worker port 冲突能被发现。
- worker workflow 文件不存在时显示 `config_invalid`，不无限重启。

# 04. 架构设计：多项目控制面

## 1. 核心结论

多项目不直接改成“一个 Orchestrator 管所有项目”。

第一版主架构：

```text
Control Plane 进程
  ├─ ProjectConfigStore
  ├─ ProjectRegistry
  ├─ ProjectProcessManager
  ├─ WorkerHealthPoller
  ├─ SummaryAggregator
  ├─ WorkflowCompiler
  └─ Phoenix Web Dashboard

Project Worker 进程 A
  └─ 原单项目 Symphony worker

Project Worker 进程 B
  └─ 原单项目 Symphony worker
```

---

## 2. 为什么不用同进程多 Orchestrator

源码复核后，当前官方实现有单项目全局依赖：

- `CLI` 的入口是一个 workflow path。
- `Workflow.workflow_file_path()` 使用 application env 中的全局 `:workflow_file_path`。
- `WorkflowStore` 用 `name: __MODULE__` 注册单例。
- `Config.settings()` 从全局 `Workflow.current()` 读取配置。
- 当前 Web Router 和 Presenter 只围绕单 orchestrator snapshot。

这些设计对单项目很合理，但对同进程多项目不友好。为了第一版稳定，不在这个层面硬改。

---

## 3. 组件设计

### 3.1 ProjectConfigStore

职责：

- 读取 `symphony.projects.yaml`。
- 校验项目配置。
- 标准化路径。
- 分配默认 logs root / pid path / worker port。
- 输出不可变项目配置。

不做：

- 不读取每个项目的完整 workflow body。
- 不接管项目 prompt/rules/hooks。

### 3.2 ProjectRegistry

职责：

- 保存控制面看到的项目列表。
- 保存每个项目当前 worker runtime 状态。
- 提供轻量查询给 Web。

状态示例：

```elixir
%ProjectRuntime{
  id: "chatgpt-extension",
  enabled: true,
  worker_status: :running,
  worker_pid: 12345,
  worker_port: 4101,
  last_seen_at: ~U[...],
  last_summary: %ProjectSummary{},
  last_error: nil
}
```

### 3.3 ProjectProcessManager

职责：

- 为每个 enabled project 启动 worker。
- 停止 worker。
- 重启 worker。
- 记录 pid。
- 捕获 worker stdout/stderr。
- 检测进程退出。

启动命令：

```bash
./bin/symphony \
  --logs-root <project.logs_root> \
  --port <project.worker_port> \
  <project.workflow_generated>
```

### 3.4 WorkerHealthPoller

职责：

- 周期性请求 worker 的 summary API。
- 超时则标记 `worker_unreachable`。
- 不主动拉取 timeline/raw。

默认周期：

```text
项目总览：2-5 秒
项目详情：1-2 秒
worker API timeout：500-1000 ms
```

### 3.5 SummaryAggregator

职责：

- 汇总多个 worker 的轻量 summary。
- 计算控制面级别 counts。
- 只输出前端总览需要的字段。

不做：

- 不合并 raw events。
- 不拉取完整 logs。
- 不解析完整 shell output。

### 3.6 WorkflowCompiler

职责：

- 按项目编译 workflow 源文件。
- 输出 `WORKFLOW.generated.md`。
- 写入源文件 hash。
- 判断 generated 文件是否 stale。

---

## 4. 进程边界

### 4.1 控制面进程

控制面负责展示和管理，不执行 Codex。

```text
Control Plane owns:
  - project config
  - worker lifecycle
  - aggregated summaries
  - dashboard routes
  - workflow compilation
```

### 4.2 worker 进程

worker 负责真实执行。

```text
Project Worker owns:
  - Linear polling
  - workspace management
  - Codex app-server session
  - retry / reconciliation
  - run trace for this project
```

### 4.3 进程隔离原则

- 一个 worker 只对应一个项目。
- 一个 worker 只读取一个 generated workflow。
- 一个 worker 只写自己的 logs root。
- 一个 worker 只绑定一个内部 API port。
- 一个 worker crash 不能带崩控制面。

## 7. 端口和安全边界

### 7.1 默认绑定

- 控制面：`127.0.0.1:4000`
- worker：`127.0.0.1:4101-4199`

说明：`4000` 作为本地常驻生产线程的默认控制面端口保留。开发、测试、手动验证或并行排查时，应改用其他空闲端口，例如 `4100`、`4123` 或 `4311`，避免端口冲突。

### 7.2 为什么 worker 也开 API

worker 自己最了解单项目运行状态。控制面如果直接读 worker 内存，会回到同进程耦合；如果只读日志，又会不可靠。worker API 是最清楚的边界。

### 7.3 暂不做认证

个人 WSL 本地环境第一版可不做复杂认证，但必须：

- 默认只绑定 loopback。
- 不把 token / secrets 展示到前端。
- raw payload 做最小脱敏。

# 05. Web 展示与 Codex 线程观测设计

## 1. 核心原则

> 后台记录足够多，前端默认展示足够少，但状态必须足够清楚。

Web 不是日志系统。总览页只回答三个问题：

1. 谁正在做事？
2. 谁可能卡住？
3. 谁需要人介入？

## 5. quiet / stalled 判断

官方 spec 已有 `codex.stall_timeout_ms`，默认 300000 ms。不要另起一套互相冲突的卡住逻辑。

建议：

```text
stall_timeout_ms = worker workflow 中的 codex.stall_timeout_ms
slow_after_ms = min(60000, stall_timeout_ms * 0.25)
quiet_after_ms = min(180000, stall_timeout_ms * 0.60)
stalled_after_ms = stall_timeout_ms
```

解释：

- `slow`：开始变慢，但仍可能正常。
- `quiet`：明显安静，应提醒观察。
- `possibly_stalled`：接近官方 stall timeout。
- `stalled`：已经达到 worker 的 stall 判定。

不要让 UI 自己计算最终卡住状态。后端 reducer 输出即可。

# 06. Workflow 模块化与生成方案

## 1. 核心原则

不取消 `WORKFLOW.md`。

改成：

```text
人维护模块化源文件
  -> workflow compiler
  -> WORKFLOW.generated.md
  -> 单项目 Symphony worker 读取 generated 文件
```

这样保留官方 workflow contract，同时降低人工维护压力。

---

## 2. 为什么不能直接取消 WORKFLOW.md

官方 spec 明确把 `WORKFLOW.md` 作为 repo-owned contract，用来承载 prompt、runtime settings、hooks、tracker config。这个设计有价值：项目策略跟代码仓库一起版本化。

如果改成控制面集中保存所有 prompt/rules/hooks，会导致：

- 项目规则脱离项目仓库；
- 多项目配置变成超级大文件；
- review 难度上升；
- AI 修改项目策略时找不到明确边界；
- 回滚困难。

所以：控制面只知道 generated workflow 的路径，不接管项目具体 prompt。

---

## 3. 源文件结构

每个项目仓库内建议：

```text
workflow/
  manifest.yaml
  frontmatter.yaml
  prompt/
    00_identity.md
    10_context.md
    20_execution_rules.md
    30_validation.md
    40_handoff.md
  skills/
    linear.md
    commit.md
    push.md
    land.md
  project/
    architecture.md
    commands.md
    conventions.md
WORKFLOW.generated.md
```

---

## 4. manifest.yaml 示例

```yaml
version: 1
output: ../WORKFLOW.generated.md
frontmatter: frontmatter.yaml
body:
  - prompt/00_identity.md
  - prompt/10_context.md
  - prompt/20_execution_rules.md
  - prompt/30_validation.md
  - prompt/40_handoff.md
include_optional:
  - skills/linear.md
  - skills/commit.md
  - skills/push.md
  - skills/land.md
  - project/architecture.md
  - project/commands.md
  - project/conventions.md
metadata:
  project_id: chatgpt-extension
  owner: local
```

## 7. 编译器职责

WorkflowCompiler 负责：

1. 读取 manifest。
2. 校验 frontmatter 是 map。
3. 校验 required fields。
4. 按顺序拼接 body 文件。
5. 生成 source hash。
6. 写入 generated 文件。
7. 返回编译报告。

不负责：

- 不解释 prompt 内容。
- 不替 Codex 决定任务策略。
- 不连接 Linear。
- 不启动 worker。

---

## 8. 编译命令

可以有两种入口：

### 8.1 mix 命令

```bash
mix workflow.compile --manifest /path/to/workflow/manifest.yaml
```

### 8.2 控制面命令

```bash
./bin/symphony_control workflow compile --project chatgpt-extension
```

第一版优先做控制面内部 compile；mix task 可后补。

---

## 9. stale 检测

控制面必须能判断：

```text
workflow source changed, generated file stale
```

依据：

- manifest hash；
- frontmatter hash；
- body file hash；
- generated 文件头部 hash。

Web 显示：

```text
workflow_status: current | stale | missing | invalid
```

worker 启动前规则：

- `missing`：不启动；
- `invalid`：不启动；
- `stale`：按配置决定自动编译或阻止启动；
- `current`：允许启动。

个人环境建议默认自动编译，但 Web 明确显示。

---

## 10. 与 Control Plane 的关系

`symphony.projects.yaml` 只保存 workflow 源和 generated 路径：

```yaml
projects:
  - id: chatgpt-extension
    workflow_source: /home/user/code/chatgpt-extension/workflow/manifest.yaml
    workflow_generated: /home/user/code/chatgpt-extension/WORKFLOW.generated.md
```

控制面不保存 prompt 内容。它只调用 compiler。

---

## 11. 质量门禁

- generated 文件能被现有 `Workflow.load()` 读取。
- generated 文件能启动现有单项目 worker。
- generated 文件包含 source hash。
- 源文件缺失时错误清晰。
- YAML 错误时错误清晰。
- 拼接顺序稳定。
- 编译器输出可复现。
- 控制面不把所有项目 prompt 加载到总览页。

---

## 12. 推荐执行顺序

Workflow 模块化不是第一阶段。它排在多项目和观测之后。

原因：

- 多项目控制面先决定项目边界。
- Web 观测先决定运行可见性。
- Workflow 编译是维护体验优化，不应阻塞核心运行链。

---

## 13. 不做的事情

第一版不做：

- 运行时动态 include。
- 复杂模板语言。
- Web 可视化 prompt 编辑器。
- 跨项目共享 prompt registry。
- 自动重写项目规则。

# 07. 质量门禁与验收标准

## 1. 总原则

这套二开最大风险不是“写不出来”，而是：

- 改坏官方单项目运行链路；
- 多项目状态串线；
- Web 默认加载过多后台信息导致卡顿；
- trace 记录太粗，仍然看不清 Codex 在干什么；
- workflow 生成物与源文件不一致。

质量门禁必须围绕这些风险设计。

---

## 2. Gate 0：证据门禁

每轮 AI 执行前必须确认：

- 已阅读相关源码文件。
- 已说明当前改动是否触碰核心执行链路。
- 已说明保留的原行为。
- 已说明新增 API 是否可能破坏旧 API。

AI 每轮输出必须包含：

```text
已检查源码：...
本轮不修改：...
本轮新增/修改：...
兼容性影响：...
```

---

## 3. Gate 1：单项目兼容门禁

任何阶段都必须保持：

```bash
./bin/symphony ./WORKFLOW.md
```

仍然可用。

验收：

- 原单项目 CLI 不因控制面变更失效。
- 原 `/api/v1/state` shape 不被破坏。
- 原 `/api/v1/<issue_identifier>` 不被破坏。
- 原 `--logs-root`、`--port` 行为不被破坏。

---

## 4. Gate 2：多项目隔离门禁

必须证明：

- 项目 A 的 workflow 不会被项目 B 读取。
- 项目 A 的 logs root 不会写到项目 B。
- 项目 A 的 workspace root 不会写到项目 B。
- 项目 A worker crash 不影响项目 B worker。
- 项目 A worker port 不与项目 B 冲突。

验收样例：

```text
启动 A、B 两个项目。
停止 A。
B 仍然 running。
A 状态变 stopped。
控制面仍 running。
```

---

## 5. Gate 3：Web 轻量展示门禁

这是硬门禁。

项目总览页不允许加载：

- raw event；
- 完整 timeline；
- 完整 prompt；
- 完整 Linear payload；
- 完整 shell output；
- 完整 tool response。

验收方式：

- 检查 LiveView assigns。
- 检查 API 请求日志。
- 检查总览页 payload size。
- 检查打开总览页时没有调用 `/raw`、`/timeline`、`/events/:id`。

通过标准：

```text
/projects 只调用 summary API。
summary payload 不包含 raw_event / payload / prompt_body / shell_output 字段。
```

---

## 6. Gate 4：Codex 状态可读性门禁

每个 run summary 必须包含：

```text
current_phase
current_action
health
last_event_at
run_duration_seconds
session_id
thread_id
turn_id
turn_count
last_error
```

至少能区分：

```text
正在准备 workspace
正在启动 Codex
Codex 正在分析
Codex 正在读取文件
Codex 正在修改文件
Codex 正在运行 shell
正在等待 shell 返回
正在等待 linear_graphql 返回
最近一段时间没有新事件
疑似卡住
正在重试
需要人工处理
```

验收：

- fake Codex events 能驱动 phase/action/health 变化。
- quiet/stalled 时间阈值可测试。
- 状态短句不依赖前端推理。

---

## 7. Gate 5：懒加载门禁

详情层必须按需加载。

验收：

- Run 详情默认最多加载最近 50 条 timeline。
- Event payload 点击后才加载。
- Raw JSON 点击后才加载。
- 大 shell output 分页。
- prompt body 默认不加载。

通过标准：

```text
打开 Run 页面时没有一次性加载全量 raw events。
打开 Event 详情时只加载该 event 的 payload。
```

---

## 8. Gate 6：Trace 可靠性门禁

Trace 系统不能影响执行主链路。

要求：

- RawEventStore 写入失败时记录 warning，不让 run 失败。
- StateReducer 异常时 fallback 到 `unknown`，不让 orchestrator crash。
- Timeline 压缩失败时保留 raw event。
- payload 超大时截断或落文件，不塞内存。

验收：

- 模拟 trace 写入目录无权限。
- 模拟超大 payload。
- 模拟 malformed event。
- worker 不应因此退出。

---

## 9. Gate 7：Workflow 编译门禁

要求：

- generated 文件能被现有 workflow loader 读取。
- source hash 与 generated hash 对得上。
- 缺文件时不启动 worker。
- YAML 错误清晰展示。
- stale 状态显示在 Web。

验收：

```text
修改 workflow/prompt/20_execution_rules.md
控制面显示 workflow stale
执行 compile
生成新的 WORKFLOW.generated.md
控制面显示 current
worker 可启动
```

---

## 10. Gate 8：性能门禁

个人 WSL 环境第一版指标：

- 1-5 个项目同时配置。
- 2 个项目同时 running。
- 总览页刷新无明显卡顿。
- summary API p95 < 500ms。
- worker API timeout 可配置。
- 单个 run raw events 上千条时，总览页性能不受明显影响。

不要求：

- 企业级 100 项目并发。
- 复杂监控指标。
- 数据库索引优化。

---

## 11. Gate 9：安全与脱敏门禁

要求：

- 默认绑定 loopback。
- 不在 Web 总览显示 API key。
- Raw payload 也要脱敏常见 secret。
- worker process command 不在 UI 展示完整 env。
- 错误日志不要泄露 token。

验收：

- 人工 grep summary response，确认无 `LINEAR_API_KEY`。
- 模拟 payload 中带 Authorization header，raw 页面显示脱敏。

---

## 12. Gate 10：回滚门禁

每阶段都必须能回滚。

要求：

- 控制面可以关闭。
- worker 可以手动按单项目方式启动。
- 新增代码失败不破坏旧命令。
- 有清理 pid / port / logs 的说明。

---

## 13. AI 每轮交付格式

AI 每轮执行结束必须输出：

```text
1. 本轮目标
2. 修改文件
3. 保持不变的旧行为
4. 新增 API / 配置
5. 测试命令
6. 测试结果
7. 通过的质量门禁
8. 未完成/已知风险
9. 回滚方式
```

阶段 1 可以不提供测试结果，但必须明确说明本轮只完成仓库初始化、文档阅读、方向确认和控制面骨架。
从阶段 2 起，必须提供与本轮验证级别匹配的验证结果：

- `L0`：说明本轮为何无需运行测试，并明确禁止补跑 `make -C elixir all`。
- `L1`：提供定向验证命令、结果、覆盖范围与未覆盖边界。
- `L2`：提供全量验证命令、结果，以及为何需要提升到全量验证。

若本轮是文档-only、只读勘测或 Linear 整理，没有测试结果是允许的，但必须有清楚的 `git diff` 判定与免测理由。

---

## 14. Definition of Done

整个二开第一版完成的定义：

- 可配置多个项目。
- 控制面能启动/停止/重启每个项目 worker。
- Web 总览能显示项目状态和 counts。
- 项目详情能显示 issue/run 的 phase/action/health。
- Run 详情能显示压缩 timeline。
- Raw event/payload/shell/prompt 均按需加载。
- Workflow 源文件能生成 `WORKFLOW.generated.md`。
- 原单项目 CLI 仍可用。
- 质量门禁 0-10 全部通过。

### 2.1 OpenAI 官方博客

来源：`https://openai.com/index/open-source-codex-orchestration-symphony/`

重要信息：

- Symphony 把 Linear 这样的项目管理板变成 coding agents 的 control plane。

- agent crash 或 stall 时，Symphony 会重启。
- OpenAI 强调从监督 Codex sessions 转向管理 work。
- 文章也指出这种工作方式带来新的问题：需要 guardrails、skills、测试和更好的文档。

对本计划的影响：

- “多项目控制面 + Web 观测”符合 Symphony 的使用方向。
- 但二开必须围绕 guardrails 和 observability，不应只堆功能。

### 2.2 GitHub README

来源：`https://github.com/openai/symphony`

重要信息：

- 仓库称 Symphony 是 low-key engineering preview，适合 trusted environments testing。
- 提供 Option 1 自己按 SPEC 实现，Option 2 使用 experimental reference implementation。
- 仓库当前没有正式 releases。

对本计划的影响：

- 二开要保守，不能假设原型机已经是稳定产品。
- 每阶段要保留回滚能力。

### 2.3 Elixir README

来源：`https://github.com/openai/symphony/blob/main/elixir/README.md`

重要信息：

- Elixir 目录是当前基于 `SPEC.md` 的实现。
- 官方明确说它是 evaluation-only prototype。
- 运行机制：poll Linear、create workspace、launch Codex app-server、send workflow prompt、keep Codex working。
- 它提供 `linear_graphql` tool。
- CLI 支持自定义 workflow path，默认 `./WORKFLOW.md`。
- `--logs-root` 控制日志目录，`--port` 启动 Phoenix observability service。
- `WORKFLOW.md` 使用 YAML front matter + Markdown body。

对本计划的影响：

- worker 进程可以继续通过 workflow path / logs-root / port 启动。
- 控制面可以利用这些已有入口管理多个 worker。

### 2.4 SPEC.md

来源：`https://github.com/openai/symphony/blob/main/SPEC.md`

重要信息：

- `Status: Draft v1 (language-agnostic)`。
- 目标包括：bounded concurrency、single authoritative orchestrator state、per-issue workspaces、terminal state cleanup、exponential backoff、repo-owned `WORKFLOW.md`、structured observability。
- Non-goals 包括：rich web UI、multi-tenant control plane、general-purpose workflow engine。
- 分层建议：Policy、Configuration、Coordination、Execution、Integration、Observability。
- Runtime snapshot 推荐包括 running、retrying、codex_totals、rate_limits。
- Dashboard/API 是 extension，不应影响 orchestrator correctness。
- Logging 要包含 issue/session context，避免大 raw payload。
- App-server client 应 emit session_started、turn_completed、turn_failed、turn_input_required、unsupported_tool_call、notification、malformed 等事件。
- Stall detection 基于 last codex timestamp 或 started_at，并使用 `codex.stall_timeout_ms`。

对本计划的影响：

- Web 增强必须是 observability/control surface，不进入 correctness path。
- 多项目必须作为外层 control plane，而不是改写核心 orchestrator 的 single authoritative state。
- Run summary 设计必须复用 snapshot/metrics，而不是让前端处理 raw events。

### 3.1 路线没有被推翻

### 3.2 多项目实现方式需要收紧

重大修正：

> 不优先做同进程多 ProjectWorker；优先做 Control Plane 管理多个单项目 OS worker。

这是因为源码全局 workflow/config/store 假设明显，直接多实例会把风险推到核心执行链路。

### 3.3 Web 观测方向正确

官方 spec 已把 dashboard/API/status surface 定为 optional extension，且要求不影响 correctness。我们的“summary 默认展示 + raw 懒加载”符合这个边界。

### 3.4 Workflow 模块化方向正确

`WORKFLOW.md` 必须保留为 generated contract。人维护源文件，worker 读取 generated 文件，兼容官方模式。

---

## 4. 重大缺陷检查

### 检查 1：是否把多项目塞进核心 Orchestrator？

结果：没有。计划改为 OS worker 隔离。

### 检查 2：是否让 Web 成为 correctness 路径？

结果：没有。Web 只读 summary/control API，不参与 orchestrator 状态决策。

### 检查 3：是否默认把 raw event 推给前端？

结果：没有。明确分 RawEventStore、TimelineStore、RunStateStore，前端默认只读 RunSummary。

### 检查 4：是否破坏 repo-owned workflow contract？

结果：没有。Workflow 仍每项目独立，generated 文件仍是官方格式。

### 检查 5：是否忽略 worker 资源开销？

结果：没有。计划承认 OS worker 开销，但以个人 WSL 少量项目为默认前提；后续出现瓶颈再评估 in-process 多 worker。

---

## 5. 保留的不确定点

这些不是阻塞项，但执行时要确认：

1. 当前 Codex app-server 事件 schema 可能随版本变化，trace normalizer 必须先读取当前 schema/实际事件样例。
2. Symphony 原型机源码可能更新，执行前应再次拉取并确认文件结构。
3. Worker API 端口如何在 WSL 和宿主浏览器之间访问，需要按你的环境实测。
4. 如果项目数量超过个人本地预期，OS worker 模式可能需要优化。
5. 如果需要强安全边界，仅 loopback 和 Codex sandbox 不一定足够，需要额外隔离。

---

## 6. 最终审核意见

可以按这套计划开工。重大架构缺陷已修正为：

```text
主线：Control Plane 管理多个单项目 OS worker
而不是：同进程直接多 Orchestrator
```

这是当前最稳路径。
