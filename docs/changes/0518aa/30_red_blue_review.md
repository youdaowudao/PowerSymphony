## 红队 Findings

红队最关键的风险点如下：

### 1. `观察层合同风险` 容易同时走向“过宽常开”和“口头绕开”

若不明确命中边界，执行层会要么把大多数任务都判成命中，要么故意把摘要、归因、多消费面说成“只是展示”，从而绕开合同路径。

已采纳动作：

- 补充四类触发条件的最小命中边界与非命中边界
- 固定为“任一命中即命中”

### 2. 角色制若无降阶矩阵，会重新膨胀成刚性全角色编制

若只列角色名而不定义适用范围，旧人数主轴会以另一种形式回来。

已采纳动作：

- 增加三类任务的最小角色适用矩阵
- 明确普通文档变更与未命中风险的普通代码变更不默认全角色常开

### 3. 角色独立性若只看 agent 名称，仍会被多窗口自审钻空

只要求“关旧开新”和“不同 agent 名称”不足以审计独立性。

已采纳动作：

- 把独立性扩成身份、线程、上下文三层约束
- 明确审计时至少要能证明角色身份不同、线程不同、后序输入不含前序完整推理历史

### 4. `contract matrix` 容易被缩成“免责窄表”

若只覆盖直接改动字段，不覆盖联动摘要、计数、归因与展示语义，checker 会被迫按残缺表过关。

已采纳动作：

- 给 `contract matrix` 增加完整性约束
- 明确 checker 与 reviewer 都有权指出 matrix 覆盖不完整
- 一旦认定不完整，必须回主线程重冻 artifact

### 5. `contract checker`、`final reviewer` 与 freeze 的仲裁顺序原先不够清楚

若不定义谁能要求重冻、谁裁决 matrix 不完整，返工链会分叉。

已采纳动作：

- 固定三者仲裁顺序
- 由主线程负责组织重冻，不允许 checker / reviewer 私自改冻结件

### 6. `baseline lock` 原先更像概念，不像协议

只有锁定对象，没有执行责任、记录字段和失效事件，会把它推向手续主义。

已采纳动作：

- 增加 `Baseline Lock` 记录协议
- 固定责任人、记录位置、最小字段、失效事件与非失效事件
- 明确 `baseline 争议` 的统一定义

### 7. `blocker ledger` 若只在返工后开启，仍会让前期争议留在口头层

风险判定争议、matrix 覆盖争议、角色独立性争议如果不入账，Workpad 仍会滞后于真实阻塞状态。

已采纳动作：

- 将 ledger 开启阈值前移到争议出现时
- 同时保留 `revise`、baseline 争议和第二轮返工作为开启条件

## 主线程裁决

裁决：**同意进入规则改写。**

理由不是“所有风险都消失了”，而是：

1. 新主轴已经成形，足以替换旧人数主轴。
2. 蓝队指出的关键定义缺口已经被回填。
3. 红队指出的主要走样路径已经被转化为成文约束。
4. 当前 change 文档已经达到“后续规则编写者不必再自行发明核心语义”的程度。

## 进入规则改写前的固定前提

后续改写 `AGENTS.md`、`elixir/WORKFLOW.md` 与治理模板时，必须保持以下定义不变：

1. 新主轴是阶段角色制，不是旧人数模板的兼容层。
2. `观察层合同风险` 是显式流程开关，且按当前命中边界执行。
3. `frozen artifact`、`contract matrix`、`contract checker`、`baseline lock`、`blocker ledger` 的定义以本 change 为准。
4. Workpad 继续是唯一活真相源。
5. 最终汇报继续以角色到位性、独立性、返工、争议、验证与放行性为主，不恢复旧的 `1+2` 主指标。
