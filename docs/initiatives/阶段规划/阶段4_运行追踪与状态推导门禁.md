# 阶段 4：运行追踪与状态推导门禁

## 阶段目标

把原始执行事件转成可读、可分页、可按需展开的运行状态，让 Web 总览只看摘要也能判断“谁在工作、谁变慢、谁需要介入”。

## 本阶段要解决的问题

- Codex / AgentRunner / Orchestrator 的原始事件过多，直接给前端会把总览页拖重。
- 只有原始事件，没有稳定的 `current_phase`、`current_action`、`health`，人很难快速判断运行状态。
- 观测层如果和执行主链路耦合过深，trace 写入失败会影响主执行。
- 需要一份阶段级门禁，明确什么叫“阶段 4 做完了”，而不是只写功能名。

## 主要产出

- `RunTrace`
- `RawEventStore`
- `TimelineStore`
- `RunStateStore`
- `EventNormalizer`
- `StateReducer`
- worker 轻量 summary API

## 必须确认

- `RawEventStore` 能看到原始事件。
- `RunStateStore` 只保留轻量 summary，不直接暴露大 payload。
- `StateReducer` 稳定输出 `current_phase`、`current_action`、`health`。
- 前端总览不直接读取 raw event。
- trace 写入失败不能让 orchestrator 崩。

## 阶段门禁

- 总览页只访问 summary 数据，不访问 raw / full timeline。
- 详情页默认只加载有限 timeline，raw 必须按需展开。
- `quiet`、`possibly_stalled`、`stalled` 的边界由后端 reducer 统一决定，不让前端自己推。
- 大 payload 必须支持折叠、截断或分页。
- 事件写入失败时，运行状态至少能降级为 `unknown`，而不是直接把整条 run 打崩。

## 质量验收口径

如果本阶段要判断“真的过关了”，至少要同时满足：

1. 原始事件可以追溯。
2. 总览页能只靠摘要判断健康度。
3. 详情页不会一次性拉爆所有原始数据。
4. 运行状态短句足够稳定，后续人不需要看完整日志也能先判断方向。
5. trace 与执行主链路分离，失败时有明确降级。
