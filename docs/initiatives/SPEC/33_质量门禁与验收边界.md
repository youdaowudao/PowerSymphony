# 质量属性与验收边界

## 作用范围

本文件只描述“多项目控制面”这个长期主题自身的质量属性和验收边界。

它不负责：

- 本仓库 agent 每轮必须执行的具体命令
- PR / CI / Linear 收口流程
- 测试分层的本仓库取值
- AI 输出格式模板

这些内容应分别维护在：

- 本仓库执行规则：`AGENTS.md`
- 可复用文档与模板规则：`docs/governance/`

## 质量属性一：单项目兼容

多项目控制面的演进不能破坏原单项目 Symphony 的使用方式。

长期要求：

- 原单项目 CLI 保持可用
- 原有单项目 API 形状不被无意破坏
- 控制面增强不能进入 orchestrator correctness 主路径

阶段收口时应能提供证据，证明兼容边界仍成立。

## 质量属性二：多项目隔离

不同项目之间必须保持运行态隔离。

长期要求：

- workflow 不串线
- logs root 不串线
- workspace root 不串线
- worker 端口不冲突
- 单个 worker crash 不带崩其他 worker 或控制面

## 质量属性三：Web 默认轻载

Web 总览层只回答“谁在工作、谁变慢、谁需要人工处理”，不默认承担调试台职责。

长期要求：

- 总览页默认只读轻量 summary
- 不默认加载 raw event、完整 timeline、完整 prompt、完整 shell output
- 详情能力通过更深层页面或显式点击进入

## 质量属性四：Codex 状态可读

人应能快速看懂当前 run 处于什么阶段、正在做什么、健康度如何。

长期要求：

- 运行摘要应稳定提供 `current_phase`、`current_action`、`health`
- 状态短句由后端推导，而不是前端自己拼装原始事件
- quiet / stalled / needs_attention 等状态具备可验证边界

## 质量属性五：详情层按需加载

控制面可以提供更深层的运行细节，但默认不一次性推给前端。

长期要求：

- Run 详情默认只加载摘要和有限 timeline
- Event payload 点击后才加载
- Raw 调试页默认分页、折叠、脱敏

## 质量属性六：Trace 不影响主执行链路

观测增强不能让执行主链路变脆。

长期要求：

- trace 写入失败不能让 run 失败
- reducer 异常时允许降级到 `unknown`
- payload 过大时允许截断、分文件或延后读取

## 质量属性七：Workflow 生成物一致性

Workflow 模块化的目标是降低维护成本，而不是引入第二套不可追踪配置。

长期要求：

- `WORKFLOW.generated.md` 始终是 worker 实际读取的合同文件
- 控制面能识别 generated 文件是否 stale
- 项目 prompt/rules/hooks 仍归项目仓库维护

## 质量属性八：性能与资源边界

第一版虽然接受 OS worker 带来的额外成本，但不能无限放大。

长期要求：

- 控制面总览页不因重数据默认加载而卡顿
- worker 轮询、详情拉取和大 payload 调试应有边界
- 项目规模增长后仍能通过项目级启停、并发和懒加载控制资源

## 质量属性九：安全与脱敏

控制面属于可信本地环境扩展，但仍需避免不必要泄露。

长期要求：

- raw 层内容仅在需要时打开
- 大 payload、prompt、shell output 支持裁剪和脱敏
- 不把调试视图当作默认首页内容

## 质量属性十：可回滚

这条长期主题中的每一类增强，都应能在必要时与官方单项目主链保持清晰边界。

长期要求：

- 控制面能力应尽量作为外层扩展而不是内嵌改坏核心执行链
- 架构、观测、workflow 生成三条线都应能单独收敛和回退

## 覆盖率门禁策略

本仓库的覆盖率策略采用“总门槛放宽、关键模块加严、逐步棘轮收紧”的长期规则。

它保护的不是“任意模块都追求同样的数字”，而是以下长期合同：

- 核心状态机和运行时 source of truth 不能因为总覆盖率达标而被稀释
- 薄壳层、UI 和框架胶水层不应为了追求汇总数字挤占核心测试预算
- `ignore_modules` 只能作为有限例外，不能演化成常态降压通道

### 总门槛

长期要求：

- 仓库总覆盖率硬门槛为 `>= 98%`
- 该总门槛用于仓库级 full gate，不替代关键模块单独门槛
- 总门槛通过，不代表关键模块自动通过

### 模块分层

覆盖率分层固定为 `Tier A`、`Tier B`、`Tier C` 三层。

#### Tier A：关键模块

命中以下任一条件时，模块应进入 `Tier A`：

- 承担状态流转或 reducer 语义
- 承担运行时 source of truth
- 承担调度、重试、恢复或继续执行语义
- 承担跨边界归因、聚合摘要或多消费面共享语义
- 承担真实副作用编排，且其错误会直接影响主执行链路

当前 `Tier A` 名单固定为：

- `SymphonyElixir.RunTrace`
- `SymphonyElixir.StateReducer`
- `SymphonyElixir.M3Precheck`
- `SymphonyElixir.ProjectProcessManager`
- `SymphonyElixir.WorkflowStore`
- `SymphonyElixir.RunStateStore`
- `SymphonyElixir.EventNormalizer`
- `SymphonyElixir.ProjectConfigStore`
- `SymphonyElixir.ProjectRegistryLoader`
- `SymphonyElixir.WorkerHealthPoller`
- `SymphonyElixir.Linear.IssueDiff`

长期目标：

- 每个 `Tier A` 模块覆盖率 `>= 99%`
- 新增 public function 不得只靠旁路集成测试顺便覆盖
- 新增分支判断不得只测 happy path

过渡期要求：

- 在所有 `Tier A` 模块都达到 `99%` 之前，任何被本次改动触达的 `Tier A` 模块，覆盖率不得低于该模块当前 baseline
- 若触达 `Tier A` 模块并新增 `case`、`cond`、`with`、重试、降级或恢复分支，应补直接命中的测试
- 过渡期结束前，不允许为了守住总门槛而降低某个 `Tier A` 模块的覆盖率

#### Tier B：业务支撑模块

`Tier B` 指有真实业务语义，但不直接承担核心状态流转或跨面合同的模块。

长期要求：

- 每个 `Tier B` 模块覆盖率目标为 `>= 97%`
- 如果某个 `Tier B` 模块开始承载状态拼装、摘要推导、跨面投影或失败语义，应升级到 `Tier A`

#### Tier C：非关键模块

`Tier C` 指以下类型模块：

- UI、LiveView、controller、presenter
- Mix task、CLI 壳层、Phoenix 框架胶水层
- 纯 schema leaf、薄适配层、静态资源壳层

长期要求：

- 每个 `Tier C` 模块覆盖率目标为 `>= 95%`
- `Tier C` 不等于“可以无测试”；它只表示测试预算不应挤占核心链路

### Diff coverage

长期要求：

- 任意触达 `elixir/**` 的 PR，新增或修改行的 diff coverage 应 `>= 90%`
- 若触达 `Tier A` 模块，该模块内 diff coverage 应 `>= 95%`
- `Tier A` 的新增 public function 必须有直接命中的测试，而不是仅依赖跨模块集成路径

这条规则是长期目标，但当前仓库还没有正式自动化脚本承载它。

在自动化脚本落地前：

- reviewer 以变更范围、定向测试和覆盖率摘要做人工复核
- 不得把“尚未自动化”理解为“不需要满足”

### ignore_modules 审计规则

`ignore_modules` 只允许作为受限例外存在。

允许进入 `ignore_modules` 的模块类型仅限：

- 纯 entrypoint 或 supervisor 壳层，且核心逻辑已下沉到可测模块
- 纯 UI 或 Phoenix 框架胶水层，几乎不承载业务分支
- 强外部依赖薄边界，当前没有稳定离线 harness，且业务语义已下沉到可测层

禁止进入 `ignore_modules` 的模块类型：

- 任意 `Tier A` 模块
- 承担状态拼装、摘要推导、计数口径、归因口径或多消费面投影的模块

每次新增、保留或移除 `ignore_modules` 条目时，必须给出以下最小审计信息：

- 模块名
- 归类理由
- 替代测试落点
- 下次复核时机

审计触发时机固定为：

- 修改 `elixir/mix.exs` 中的 `ignore_modules`
- 调整模块 tier
- 阶段收口前复核覆盖率策略时

如果一个被 ignore 的模块开始承载真实业务分支、状态推导或跨面语义，应优先把逻辑拆走；若无法拆走，则必须移出 `ignore_modules` 并补测试。

### 自动化脚本边界

覆盖率分层校验与 diff coverage 校验最终应由本地脚本或 CI 自动化承载，但自动化本身不得破坏现有验证分层。

长期约束：

- 不得把新的覆盖率脚本变成绕过 `Next Push Gate` 的平行入口
- 不得改变 `make all` 仍以 `mix test --cover` 为仓库总覆盖率真相源的事实
- 不得让脚本在本地和 CI 采用不同阈值口径
- 不得把人工 reviewer 才能判断的 tier 归类悄悄写死成难以复核的隐式逻辑

在脚本方案被明确设计、复核并接入前：

- 本文档先定义长期策略
- 脚本落地单独作为后续变更处理
- 若脚本设计与现有 `make all`、`Next Push Gate`、`ignore_modules` 或 coverage 汇总口径冲突，必须先回到人类裁决，不得直接接入
