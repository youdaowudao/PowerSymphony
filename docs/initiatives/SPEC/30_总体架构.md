# 多项目控制面的总体架构

## 1. 核心结论

多项目不直接改成“一个 Orchestrator 管所有项目”。

第一版主架构：

```text
Control Plane 进程
  ├─ ProjectConfigStore
  ├─ ProjectRegistry
  ├─ ProjectProcessManager
  ├─ WorkerHealthPoller
  ├─ SummaryAggregator
  ├─ WorkflowCompiler
  └─ Phoenix Web Dashboard

Project Worker 进程 A
  └─ 原单项目 Symphony worker

Project Worker 进程 B
  └─ 原单项目 Symphony worker
```

---

## 2. 为什么不用同进程多 Orchestrator

源码复核后，当前官方实现有单项目全局依赖：

- `CLI` 的入口是一个 workflow path。
- `Workflow.workflow_file_path()` 使用 application env 中的全局 `:workflow_file_path`。
- `WorkflowStore` 用 `name: __MODULE__` 注册单例。
- `Config.settings()` 从全局 `Workflow.current()` 读取配置。
- 当前 Web Router 和 Presenter 只围绕单 orchestrator snapshot。

这些设计对单项目很合理，但对同进程多项目不友好。为了第一版稳定，不在这个层面硬改。

## 3. 组件设计

### 3.1 ProjectConfigStore

职责：

- 读取 `symphony.projects.yaml`。
- 校验项目配置。
- 标准化路径。
- 分配默认 logs root / pid path / worker port。
- 输出不可变项目配置。

不做：

- 不读取每个项目的完整 workflow body。
- 不接管项目 prompt/rules/hooks。

### 3.2 ProjectRegistry

职责：

- 保存控制面看到的项目列表。
- 保存每个项目当前 worker runtime 状态。
- 提供轻量查询给 Web。

状态示例：

```elixir
%ProjectRuntime{
  id: "chatgpt-extension",
  enabled: true,
  worker_status: :running,
  worker_pid: 12345,
  worker_port: 4101,
  last_seen_at: ~U[...],
  last_summary: %ProjectSummary{},
  last_error: nil
}
```

### 3.3 ProjectProcessManager

职责：

- 为每个 enabled project 启动 worker。
- 停止 worker。
- 重启 worker。
- 记录 pid。
- 捕获 worker stdout/stderr。
- 检测进程退出。

启动命令：

```bash
./bin/symphony \
  --logs-root <project.logs_root> \
  --port <project.worker_port> \
  <project.workflow_generated>
```

### 3.4 WorkerHealthPoller

职责：

- 周期性请求 worker 的 summary API。
- 超时则标记 `worker_unreachable`。
- 不主动拉取 timeline/raw。

默认周期：

```text
项目总览：2-5 秒
项目详情：1-2 秒
worker API timeout：500-1000 ms
```

### 3.5 SummaryAggregator

职责：

- 汇总多个 worker 的轻量 summary。
- 计算控制面级别 counts。
- 只输出前端总览需要的字段。

不做：

- 不合并 raw events。
- 不拉取完整 logs。
- 不解析完整 shell output。

### 3.6 WorkflowCompiler

职责：

- 按项目编译 workflow 源文件。
- 输出 `WORKFLOW.generated.md`。
- 写入源文件 hash。
- 判断 generated 文件是否 stale。

## 4. 进程边界

### 4.1 控制面进程

控制面负责展示和管理，不执行 Codex。

```text
Control Plane owns:
  - project config
  - worker lifecycle
  - aggregated summaries
  - dashboard routes
  - workflow compilation
```

### 4.2 worker 进程

worker 负责真实执行。

```text
Project Worker owns:
  - Linear polling
  - workspace management
  - Codex app-server session
  - retry / reconciliation
  - run trace for this project
```

### 4.3 进程隔离原则

- 一个 worker 只对应一个项目。
- 一个 worker 只读取一个 generated workflow。
- 一个 worker 只写自己的 logs root。
- 一个 worker 只绑定一个内部 API port。
- 一个 worker crash 不能带崩控制面。

## 5. API 设计

### 5.1 控制面 API

```text
GET  /api/v1/projects
GET  /api/v1/projects/:project_id/summary
POST /api/v1/projects/:project_id/start
POST /api/v1/projects/:project_id/stop
POST /api/v1/projects/:project_id/restart
POST /api/v1/projects/:project_id/refresh
GET  /api/v1/projects/:project_id/runs/:run_id
GET  /api/v1/projects/:project_id/runs/:run_id/timeline
GET  /api/v1/projects/:project_id/runs/:run_id/events/:event_id
GET  /api/v1/projects/:project_id/runs/:run_id/raw
```

控制面详情 API 可以代理到 worker，但必须保留懒加载边界。

### 5.2 worker API 扩展

保留官方已有：

```text
GET  /api/v1/state
GET  /api/v1/<issue_identifier>
POST /api/v1/refresh
```

新增：

```text
GET /api/v1/runs/summary
GET /api/v1/runs/:run_id
GET /api/v1/runs/:run_id/timeline?limit=50&cursor=...
GET /api/v1/runs/:run_id/events/:event_id
GET /api/v1/runs/:run_id/raw?section=events|prompt|shell|linear&cursor=...
```

### 5.3 API envelope

成功：

```json
{
  "generated_at": "2026-05-03T12:00:00Z",
  "data": {}
}
```

错误：

```json
{
  "error": {
    "code": "worker_unavailable",
    "message": "Project worker did not respond within 1000ms"
  }
}
```

## 7. 端口和安全边界

### 7.1 默认绑定

- 控制面：`127.0.0.1:4000`
- worker：`127.0.0.1:4101-4199`

说明：`4000` 作为本地常驻生产线程的默认控制面端口保留。开发、测试、手动验证或并行排查时，应改用其他空闲端口，例如 `4100`、`4123` 或 `4311`，避免端口冲突。

### 7.2 为什么 worker 也开 API

worker 自己最了解单项目运行状态。控制面如果直接读 worker 内存，会回到同进程耦合；如果只读日志，又会不可靠。worker API 是最清楚的边界。

### 7.3 暂不做认证

个人 WSL 本地环境第一版可不做复杂认证，但必须：

- 默认只绑定 loopback。
- 不把 token / secrets 展示到前端。
- raw payload 做最小脱敏。

## 9. 回滚策略

任何阶段都必须能回滚到：

```bash
./bin/symphony ./WORKFLOW.md --port 4100
```

这里的 `4100` 是本地开发/测试回滚验证示例端口；如果本机 `4000` 已被常驻生产线程占用，不要复用 `4000` 做临时验证。

控制面改坏时，不应该影响单项目 worker 的手动启动能力。

## 10. 未来可选但第一版不做

- 同进程多 Orchestrator。
- 全局精确 agent concurrency scheduler。
- 控制面写入 Linear。
- 远程部署和身份认证。
- 数据库化事件存储。
- 前端复杂筛选/图表。
