# 03. 总体计划与里程碑

## 1. 总体路线图

```text
阶段 1：创建仓库、阅读项目文档、确认方向与控制面骨架
阶段 2：Project Worker 进程管理
阶段 3：轻量 Web 总览与项目详情
阶段 4：Codex Run Trace 与状态推导
阶段 5：按需加载的详情页与 raw 调试页
阶段 6：Workflow 模块化生成
阶段 7：硬化、回归、使用文档
```

注意：这些是阶段，不是小任务。后续 AI 可以在每个阶段内部拆 task。

---

## 2. 阶段 1：创建仓库、阅读项目文档、确认方向与控制面骨架

### 目标

完成项目初始化和第一版控制面骨架，为后续多项目扩展建立稳定入口。

### 主要产出

- 本地仓库创建完成，并已同步远端主线。
- 已阅读 `DOCS/` 中的项目规划文档，并确认主线方向。
- `symphony.projects.yaml` schema。
- 至少 2 个项目的配置样例。
- `ProjectConfig` 解析与校验。
- `ProjectRegistry` 内存状态。
- 新 CLI：`./bin/symphony_control --config ./symphony.projects.yaml --port 4000`。
- 控制面 Web/API 能显示项目列表、配置校验结果和基础配置态/占位态运行状态。

### 必须确认

- 控制面至少能管理 2 个项目。
- 每个项目有独立 `workflow_generated`、`workspace_root`、`logs_root` 和运行状态占位字段。
- 项目方向以本套规划文档为准，不偏离多项目 Control Plane 主线。
- 原单项目 CLI 不被破坏。
- 阶段 1 的状态仅限配置态、校验态和 `not_started` 一类占位态，不提前实现真实 worker 生命周期。

### M1 基线冻结说明

作为 `C-11` 的冻结结论，阶段 1 后续事项统一按以下边界执行：

- M1 范围：仓库初始化与远端同步、文档基线确认、`symphony.projects.yaml` 静态配置模型、`ProjectRegistry` 占位状态、`symphony_control` CLI / 控制面骨架、阶段门禁。
- M1 非范围：真实 worker 启停、健康检查、stdout / stderr 日志落盘、Codex 线程事件采集、Trace / RawEventStore、Workflow 模块化生成、同进程多 Orchestrator、重写官方核心执行链路。
- M2 起点：只有进入阶段 2 时，才开始 Project Worker OS 进程生命周期管理，以及相关健康检查与日志落盘。
- 兼容要求：整个 M1 期间必须持续保持 `./bin/symphony ./WORKFLOW.md` 可用。
- 执行参考：主参考文档集合以 `00_README_执行入口.md` 中的 “M1 必读参考集合” 为准。

### 阶段门禁

- 配置错误能清晰展示。
- 项目 ID 校验防止路径穿越和重复。
- 控制面至少能管理 2 个项目。
- 各项目 `workflow`、`workspace`、日志和状态占位字段不串线。
- 不影响原单项目 CLI。
- 对存在未完成前置依赖的 issue，调度器不能仅因其处于 active state 就直接派发。
- 真实 worker 启停、健康检查、stdout/stderr 落盘统一放到阶段 2。

### M1-5：前置依赖识别与自动阻塞基线修正

#### 目标

修正当前单项目 Symphony worker 的调度基线：当 issue 仍存在未完成前置依赖时，即使该 issue 已被人工置为 `In Progress` 一类 active state，也不能被调度器直接派发执行。

#### 为什么放在 M1

- 这不是临时 bug 备注，而是后续多项目控制面必须依赖的调度语义基线。
- 如果不先修正，后续阶段一旦开始更依赖 Symphony 自主推进 issue，仍会退回到“人工维护下游长期停在 `Todo`”的脆弱流程。
- 阶段 2 会开始真正管理多个项目 worker；在那之前，必须先明确单项目 worker 的依赖阻塞语义。

#### 主要产出

- 当前调度行为与目标行为的文档化说明。
- 对 `blocked_by` 的统一调度规则：不只拦 `Todo`，而是拦所有可执行 active issue。
- 首次派发与 retry 派发使用同一套依赖阻塞判断。
- 针对依赖链顺序的最小验证用例。

#### 必须确认

- 不能依赖人工长期维持下游 issue 停在 `Todo`。
- 调度器必须自己判断前置依赖是否全部进入 terminal state。
- 依赖阻塞规则属于执行正确性，不属于 Web 展示层逻辑。
- 该修正不改变项目方向，不引入复杂 DAG 引擎。

#### 阶段门禁补充

- 上游 issue 未完成、下游 issue 已在 `In Progress` 时，下游仍不能被派发。
- 上游 issue 进入 terminal state 后，下游 issue 才重新具备可调度资格。
- retry 路径与首次派发路径对依赖阻塞的判断一致。

### M1-6：`turn/completed` 与错误续跑语义修正

#### 目标

修正单项目 Symphony worker 在 `turn/completed`、run 结束判定与 active issue continuation 上的基线语义，避免“turn 正常结束但 ticket 未闭环”被错误放大为连续续跑。

#### 为什么放在 M1

- 这不是单纯日志文案问题，而是单项目执行链的正确性问题。
- 如果不先修正，后续多项目控制面会建立在错误的 turn/run 边界上。
- 阶段 2 之前必须先冻结单项目 worker 的 continuation 语义，不然后续 worker 生命周期和调度策略会继续放大这类误续跑。

#### 主要产出

- turn 级完成与 run 级完成的语义拆层说明。
- active issue 下错误续跑边界的规划结论。
- `AgentRunner` 与 `Orchestrator` 的 run 级结果边界说明。
- `max_turns` 之后合法 continuation 路径的保留原则。
- 对应专题参考文档集合：`DOCS/plan_rerun_fix/`。

#### 必须确认

- `turn/completed` 仍然只是协议层的 turn 级终态，不冒充 ticket 完成。
- 主修点在 `agent_runner.ex` / `orchestrator.ex` 边界，不在 `app_server.ex` 协议接收本身。
- 外层 continuation 不能简单删除，因为它还承担 `max_turns` 打满后的合法续跑职责。
- 后续实现若要引入 run 级结构化终态，必须同时设计 `AgentRunner` 到 `Orchestrator` 的结果上报机制。

#### 阶段门禁补充

- `turn/completed` 不再被系统外观或编排逻辑误读为任务完成。
- 对 active issue，turn 级完成不能自动升级成健康的 run 完成。
- 错误收口必须与 timeout / crash / success 分开处理。
- 兼容性上继续保持原单项目 CLI 与 `./bin/symphony ./WORKFLOW.md` 不被破坏。

### M1-7：M1final 门禁收口与兼容性验收

#### 目标

对 M1 做阶段性收口，验证 M1 的计划边界、兼容性、配置读取、项目占位展示和最小 smoke check 都成立，为进入阶段 2 做清晰交接。

#### 为什么放在 M1 最后

- 这张卡不是补功能，而是防止 M1 在边界未对齐时“假完成”。
- 当前 M1 已不只包含最初的配置和控制面骨架，还补入了两项执行语义基线修正，因此需要单独的最终门禁项统一收口。
- 没有这一步，阶段 2 开始时很容易重新争论哪些边界已经冻结、哪些仍未完成。

#### 主要产出

- M1 已交付清单。
- M1 保持不变的旧行为清单。
- M1 已通过的门禁项清单。
- M1 明确未做项与进入阶段 2 的交接说明。
- 最小 smoke check 结果。

#### 必须确认

- M1 不要求完成真实 Linear / Codex / worker E2E。
- M1 必须保持原单项目 CLI 可用。
- 控制面至少能管理 2 个项目，并且只展示配置态、校验态和占位态。
- 真实 worker 启停、健康检查、stdout/stderr 落盘仍然统一放到阶段 2。

#### 阶段门禁补充

- 验证 `./bin/symphony ./WORKFLOW.md` 入口未被破坏。
- 验证 `symphony_control` 能读取样例配置并展示至少 2 个项目的占位态。
- 验证本阶段未提前引入真实 worker 生命周期能力。
- 输出进入阶段 2 前的明确 handoff。

## 3. 阶段 2：Project Worker 进程管理

### 目标

控制面能管理多个单项目 Symphony OS worker。

### 主要产出

- `ProjectProcessManager`。
- worker 启动命令生成。
- worker 健康检查。
- 启动/停止/重启 API。
- worker stdout/stderr 文件落盘。
- worker crash 后状态可见。

### worker 启动模式

每个项目启动为独立 OS 进程：

```bash
./bin/symphony \
  --logs-root <project.logs_root> \
  --port <project.worker_port> \
  <project.workflow_generated>
```

控制面只绑定主 Web 端口，例如 `4000`。worker 端口只绑定 loopback，例如 `127.0.0.1:4101`。

### 关键设计

- 控制面保存 pid 文件。
- worker crash 不让控制面 crash。
- worker API 不直接暴露给外网。
- worker logs root 与 workspace root 均按项目隔离。
- 控制面不要解析 worker raw log 来判断主状态；优先调用 worker summary API。

### 阶段门禁

- 同时启动两个 dummy worker。
- 停止一个 worker 不影响另一个。
- worker port 冲突能被发现。
- worker workflow 文件不存在时显示 `config_invalid`，不无限重启。

## 4. 阶段 3：轻量 Web 总览与项目详情

### 目标

让用户能在 Web 上看到多项目整体状态，但页面绝不加载重数据。

### 主要产出

- `/projects` 总览页。
- `/projects/:project_id` 项目详情页。
- `/api/v1/projects`。
- `/api/v1/projects/:project_id/summary`。
- summary 聚合器。
- summary diff 推送机制。

### 总览页字段

```text
project_id
name
enabled
worker_status
running_count
quiet_count
possibly_stalled_count
stalled_count
retrying_count
needs_attention_count
last_activity_at
last_error
worker_port
```

### 项目详情页字段

```text
issue_identifier
title
linear_state
current_phase
current_action
health
session_id
thread_id
turn_id
turn_count
last_event_at
run_duration_seconds
last_error
```

### 阶段门禁

- 浏览总览页时不访问 worker timeline/raw endpoints。
- 浏览总览页时不加载完整 prompt、payload、shell output。
- LiveView assigns 不包含 raw event 列表。
- 页面刷新频率可控。

## 5. 阶段 4：Codex Run Trace 与状态推导

### 目标

让单项目 worker 能把 Codex 原始事件转成可读的 run 状态。

### 主要产出

- `RunTrace`。
- `RawEventStore`。
- `RunStateStore`。
- `TimelineStore`。
- `EventNormalizer`。
- `StateReducer`。
- worker 新增轻量 summary API。

### 数据流

```text
Codex / AgentRunner / Orchestrator / Linear tool / Hook
  -> RawEventStore
  -> EventNormalizer
  -> StateReducer
  -> RunStateStore
  -> Summary API
```

### 阶段门禁

- `RawEventStore` 中可看到原始事件。
- `RunStateStore` 中只有轻量 summary。
- `StateReducer` 能输出 `current_phase/current_action/health`。
- UI 不直接依赖 raw event。
- 事件写入失败不让 orchestrator 崩。

## 6. 阶段 5：按需加载详情页与 raw 调试页

### 目标

提供足够排错能力，但严格按需加载。

### 主要产出

- `/projects/:project_id/runs/:run_id` Run 详情页。
- `/projects/:project_id/runs/:run_id/timeline`。
- `/projects/:project_id/runs/:run_id/events/:event_id`。
- `/projects/:project_id/runs/:run_id/raw`。
- timeline 分页。
- raw payload 懒加载。

### 默认加载策略

- Run 详情页默认加载最近 50 条压缩事件。
- Event detail 点击后才加载 payload 摘要。
- Raw JSON 点击后才加载完整原始数据。
- shell output 默认折叠，只展示摘要和最后几行。
- prompt 只显示 prompt hash、生成时间、长度；展开才加载。

### 阶段门禁

- 打开项目页不会加载 timeline。
- 打开 Run 页不会加载所有 raw payload。
- 打开 Event detail 只加载单个 event payload。
- 大 payload 有大小限制和分页。

### 可供扩展讨论

后续在不改变“前端总览只看重要事项”这一原则的前提下，Run 详情层应保留继续扩展为“完整会话研究视图”的可能：在明确需要时，允许用户点击进入某个 run / thread 后，尽可能完整地查看该次执行的详细实情，包括完整对话、工具调用、子 Agent 调用、shell/output 摘要，以及与这些内容关联的必要原始事件数据。该能力先作为愿景保留，后续再单独细化数据分层、实时策略、脱敏边界与前端呈现方式。

## 7. 阶段 6：Workflow 模块化生成

### 目标

减轻 `WORKFLOW.md` 维护压力，同时保留官方 workflow contract。

### 主要产出

- `workflow/manifest.yaml`。
- `workflow/*.md` 源文件结构。
- `mix workflow.compile` 或控制面 compile 功能。
- `WORKFLOW.generated.md`。
- stale 检测。
- Web 显示 workflow hash/status。

### 生成原则

- 人只维护源文件。
- worker 只读取 generated 文件。
- generated 文件带 source hash。
- 生成失败不启动 worker。
- 不把所有项目配置塞进一个 workflow。

### 阶段门禁

- 源文件变更后可重新生成。
- generated 文件语法符合官方 front matter + Markdown body。
- 生成前后 hash 可追踪。
- 旧的手写 `WORKFLOW.md` 模式仍兼容。

## 8. 阶段 7：硬化、回归、文档

### 目标

把前面功能从“能跑”变成“能长期用”。

### 主要产出

- 回归测试套件。
- 性能测试记录。
- 故障注入测试。
- 使用手册。
- AI 维护手册。
- 升级/回滚说明。

### 硬化重点

- worker crash recovery。
- worker port collision。
- stale workflow。
- bad YAML。
- invalid project id。
- runaway raw event growth。
- trace file rotate。
- LiveView 卡顿检测。
- worker API timeout。

### 阶段门禁

- 两项目并发运行 30 分钟，控制面可用。
- 一个 worker crash，另一个不受影响。
- 大量 raw event 不让总览页变慢。
- 所有 raw 数据都有懒加载入口。
- 原单项目 CLI 行为未破坏。
