# 03. 总体计划与里程碑

## 1. 总体路线图

```text
阶段 1：创建仓库、阅读项目文档、确认方向与控制面骨架
阶段 2：Project Worker 进程管理
阶段 3：轻量 Web 总览与项目详情
阶段 4：Codex Run Trace 与状态推导
阶段 5：按需加载的详情页与 raw 调试页
阶段 6：Workflow 模块化生成
阶段 7：硬化、回归、使用文档
```

注意：这些是阶段，不是小任务。后续 AI 可以在每个阶段内部拆 task。

---

## 2. 阶段 1：创建仓库、阅读项目文档、确认方向与控制面骨架

### 目标

完成项目初始化和第一版控制面骨架，为后续多项目扩展建立稳定入口。

### 主要产出

- 本地仓库创建完成，并已同步远端主线。
- 已阅读 `DOCS/` 中的项目规划文档，并确认主线方向。
- `symphony.projects.yaml` schema。
- 至少 2 个项目的配置样例。
- `ProjectConfig` 解析与校验。
- `ProjectRegistry` 内存状态。
- 新 CLI：`./bin/symphony_control --config ./symphony.projects.yaml --port 4000`。
- 控制面 Web/API 能显示项目列表、配置校验结果和基础配置态/占位态运行状态。

### 必须确认

- 控制面至少能管理 2 个项目。
- 每个项目有独立 `workflow_generated`、`workspace_root`、`logs_root` 和运行状态占位字段。
- 项目方向以本套规划文档为准，不偏离多项目 Control Plane 主线。
- 原单项目 CLI 不被破坏。
- 阶段 1 的状态仅限配置态、校验态和 `not_started` 一类占位态，不提前实现真实 worker 生命周期。

### 阶段门禁

- 配置错误能清晰展示。
- 项目 ID 校验防止路径穿越和重复。
- 控制面至少能管理 2 个项目。
- 各项目 `workflow`、`workspace`、日志和状态占位字段不串线。
- 不影响原单项目 CLI。
- 真实 worker 启停、健康检查、stdout/stderr 落盘统一放到阶段 2。

## 3. 阶段 2：Project Worker 进程管理

### 目标

控制面能管理多个单项目 Symphony OS worker。

### 主要产出

- `ProjectProcessManager`。
- worker 启动命令生成。
- worker 健康检查。
- 启动/停止/重启 API。
- worker stdout/stderr 文件落盘。
- worker crash 后状态可见。

### worker 启动模式

每个项目启动为独立 OS 进程：

```bash
./bin/symphony \
  --logs-root <project.logs_root> \
  --port <project.worker_port> \
  <project.workflow_generated>
```

控制面只绑定主 Web 端口，例如 `4000`。worker 端口只绑定 loopback，例如 `127.0.0.1:4101`。

### 关键设计

- 控制面保存 pid 文件。
- worker crash 不让控制面 crash。
- worker API 不直接暴露给外网。
- worker logs root 与 workspace root 均按项目隔离。
- 控制面不要解析 worker raw log 来判断主状态；优先调用 worker summary API。

### 阶段门禁

- 同时启动两个 dummy worker。
- 停止一个 worker 不影响另一个。
- worker port 冲突能被发现。
- worker workflow 文件不存在时显示 `config_invalid`，不无限重启。

## 4. 阶段 3：轻量 Web 总览与项目详情

### 目标

让用户能在 Web 上看到多项目整体状态，但页面绝不加载重数据。

### 主要产出

- `/projects` 总览页。
- `/projects/:project_id` 项目详情页。
- `/api/v1/projects`。
- `/api/v1/projects/:project_id/summary`。
- summary 聚合器。
- summary diff 推送机制。

### 总览页字段

```text
project_id
name
enabled
worker_status
running_count
quiet_count
possibly_stalled_count
stalled_count
retrying_count
needs_attention_count
last_activity_at
last_error
worker_port
```

### 项目详情页字段

```text
issue_identifier
title
linear_state
current_phase
current_action
health
session_id
thread_id
turn_id
turn_count
last_event_at
run_duration_seconds
last_error
```

### 阶段门禁

- 浏览总览页时不访问 worker timeline/raw endpoints。
- 浏览总览页时不加载完整 prompt、payload、shell output。
- LiveView assigns 不包含 raw event 列表。
- 页面刷新频率可控。

## 5. 阶段 4：Codex Run Trace 与状态推导

### 目标

让单项目 worker 能把 Codex 原始事件转成可读的 run 状态。

### 主要产出

- `RunTrace`。
- `RawEventStore`。
- `RunStateStore`。
- `TimelineStore`。
- `EventNormalizer`。
- `StateReducer`。
- worker 新增轻量 summary API。

### 数据流

```text
Codex / AgentRunner / Orchestrator / Linear tool / Hook
  -> RawEventStore
  -> EventNormalizer
  -> StateReducer
  -> RunStateStore
  -> Summary API
```

### 阶段门禁

- `RawEventStore` 中可看到原始事件。
- `RunStateStore` 中只有轻量 summary。
- `StateReducer` 能输出 `current_phase/current_action/health`。
- UI 不直接依赖 raw event。
- 事件写入失败不让 orchestrator 崩。

## 6. 阶段 5：按需加载详情页与 raw 调试页

### 目标

提供足够排错能力，但严格按需加载。

### 主要产出

- `/projects/:project_id/runs/:run_id` Run 详情页。
- `/projects/:project_id/runs/:run_id/timeline`。
- `/projects/:project_id/runs/:run_id/events/:event_id`。
- `/projects/:project_id/runs/:run_id/raw`。
- timeline 分页。
- raw payload 懒加载。

### 默认加载策略

- Run 详情页默认加载最近 50 条压缩事件。
- Event detail 点击后才加载 payload 摘要。
- Raw JSON 点击后才加载完整原始数据。
- shell output 默认折叠，只展示摘要和最后几行。
- prompt 只显示 prompt hash、生成时间、长度；展开才加载。

### 阶段门禁

- 打开项目页不会加载 timeline。
- 打开 Run 页不会加载所有 raw payload。
- 打开 Event detail 只加载单个 event payload。
- 大 payload 有大小限制和分页。

## 7. 阶段 6：Workflow 模块化生成

### 目标

减轻 `WORKFLOW.md` 维护压力，同时保留官方 workflow contract。

### 主要产出

- `workflow/manifest.yaml`。
- `workflow/*.md` 源文件结构。
- `mix workflow.compile` 或控制面 compile 功能。
- `WORKFLOW.generated.md`。
- stale 检测。
- Web 显示 workflow hash/status。

### 生成原则

- 人只维护源文件。
- worker 只读取 generated 文件。
- generated 文件带 source hash。
- 生成失败不启动 worker。
- 不把所有项目配置塞进一个 workflow。

### 阶段门禁

- 源文件变更后可重新生成。
- generated 文件语法符合官方 front matter + Markdown body。
- 生成前后 hash 可追踪。
- 旧的手写 `WORKFLOW.md` 模式仍兼容。

## 8. 阶段 7：硬化、回归、文档

### 目标

把前面功能从“能跑”变成“能长期用”。

### 主要产出

- 回归测试套件。
- 性能测试记录。
- 故障注入测试。
- 使用手册。
- AI 维护手册。
- 升级/回滚说明。

### 硬化重点

- worker crash recovery。
- worker port collision。
- stale workflow。
- bad YAML。
- invalid project id。
- runaway raw event growth。
- trace file rotate。
- LiveView 卡顿检测。
- worker API timeout。

### 阶段门禁

- 两项目并发运行 30 分钟，控制面可用。
- 一个 worker crash，另一个不受影响。
- 大量 raw event 不让总览页变慢。
- 所有 raw 数据都有懒加载入口。
- 原单项目 CLI 行为未破坏。
