# 08. AI 执行手册与交付提示词

## 1. 总执行提示词

把下面这段作为后续每轮 AI 执行的开头：

```text
你正在基于 OpenAI Symphony Elixir 原型机做二次开发。禁止换语言，禁止全量重写，禁止把多个项目塞进一个核心 Orchestrator。主线是：新增多项目 Control Plane，管理多个单项目 Symphony worker OS 进程；每个 worker 仍读取自己的 WORKFLOW.generated.md、logs_root、workspace_root、内部 API port。Web 总览页只能读取轻量 summary，不允许默认拉取 raw event、完整 timeline、完整 prompt、完整 shell output。Codex 线程观测必须通过后端 StateReducer 生成 current_phase/current_action/health，再给前端展示。每轮修改必须先按当前阶段要求执行：阶段 1 先完成创建仓库、阅读文档、确认方向和骨架搭建；从阶段 2 起必须先读当前源码和相关测试，并先按《07_质量门禁与验收标准.md》基于 `git diff` 判断验证级别，再给出设计差异、修改文件、与本轮验证级别匹配的验证命令或免测理由、回滚方式。禁止默认运行 `make -C elixir all`；只有核心代码改动、测试/构建配置改动、或 PR 前最终重核验时才允许。文档-only、只读勘测、Linear 整理阶段不得运行该命令。不要生成换语言方案，不要提前拆小型任务列表，按已定阶段和质量门禁推进。
```

## 2. 阶段执行模板

### 阶段 1 提示词

```text
请执行阶段 1：创建仓库、阅读项目文档、确认方向与控制面骨架。完成仓库初始化和远端主线同步，阅读 `DOCS/` 中的规划文档并确认方向；新增 `symphony.projects.yaml` schema、ProjectConfigStore、ProjectRegistry 和 `symphony_control` CLI 骨架。控制面至少能管理 2 个项目，并为每个项目保存独立的 workflow、workspace、日志和状态占位字段。阶段 1 只允许实现配置态、校验态和 `not_started` 一类占位态，不要提前实现真实 worker 启停、健康检查和日志落盘。阶段 1 的后半段还包含两项执行语义基线修正：前置依赖识别与自动阻塞、`turn/completed` 与错误续跑语义修正；最后必须经过 M1final 门禁收口与兼容性验收。暂时不要提前做 Codex trace UI，必须保持原 `./bin/symphony ./WORKFLOW.md` 可用。
```

### 阶段 2 提示词

```text
请执行阶段 2：Project Worker 进程管理。控制面通过 OS 进程启动多个单项目 Symphony worker，每个 worker 有独立 workflow_generated、logs_root、worker_port。实现 start/stop/restart/health 基础能力。不要做同进程多 Orchestrator。必须验证停止一个 worker 不影响另一个 worker。
```

### 阶段 3 提示词

```text
请执行阶段 3：轻量 Web 总览与项目详情。新增 /projects 和 /projects/:project_id 页面，以及 summary API。总览页只能读取 ProjectSummary/RunSummary，禁止加载 timeline/raw/prompt/shell output。必须证明 LiveView assigns 中没有 raw event。
```

### 阶段 4 提示词

```text
请执行阶段 4：Codex Run Trace 与状态推导。请在单项目 worker 内新增 RawEventStore、EventNormalizer、StateReducer、RunStateStore、TimelineStore 的最小可用版本。输出 current_phase/current_action/health。Trace 写入失败不能影响 orchestrator correctness。不要让前端直接推理 raw event。
```

### 阶段 5 提示词

```text
请执行阶段 5：按需加载详情页与 raw 调试页。新增 Run 详情页、timeline 分页、event detail 懒加载、raw 调试页。默认只加载最近 50 条压缩 timeline。payload、prompt、shell output 必须点击后才加载，并有大小限制/分页/脱敏。
```

### 阶段 6 提示词

```text
请执行阶段 6：Workflow 模块化生成。实现 workflow/manifest.yaml + frontmatter.yaml + 多个 prompt 源文件生成 WORKFLOW.generated.md。保留官方 front matter + Markdown body 格式。控制面显示 workflow current/stale/missing/invalid 状态。不要把所有项目 prompt 放进 symphony.projects.yaml。
```

### 阶段 7 提示词

```text
请执行阶段 7：硬化、回归、使用文档。做 worker crash、bad YAML、port collision、large payload、trace write failure、worker API timeout、stale workflow 等故障注入测试。输出最终使用手册和回滚说明。
```

## 3. AI 每轮必须先回答的问题

在动代码前，AI 必须先回答：

```text
0. 本轮 `git diff` 判断下的验证级别是什么，为什么？
1. 本轮是否会修改核心 Orchestrator / AgentRunner / Codex AppServer？如果会，为什么必须？
2. 本轮如何保持单项目启动方式不变？
3. 本轮是否会让前端加载更多后台数据？如果会，为什么不可避免？
4. 本轮新增状态是否由后端 reducer 推导？
5. 本轮的回滚方式是什么？
```

如果 AI 回答不清楚，不要让它改代码。

---

## 4. AI 禁止事项

明确禁止：

- 提出换语言路线。
- 重写整个 Symphony。
- 直接把多个 workflow 塞进一个 `WORKFLOW.md`。
- 第一版去全局化所有核心模块。
- 让前端订阅 raw Codex event。
- 在 LiveView assigns 放完整 payload。
- 把 prompt body 默认展示在总览页。
- 为了 UI 便利修改 orchestrator 正确性逻辑。
- 在需要 `L1` 或 `L2` 验证的轮次里，没有测试结果就声称完成。
- 在 `L0` 轮次里跳过 `git diff` 判定，或擅自补跑 `make -C elixir all`。

## 5. AI 推荐工作方式

每轮让 AI 采用这个节奏：

```text
读取源码
  -> 查看 git diff
  -> 判断验证级别（L0 / L1 / L2）
  -> 输出差异分析
  -> 给出本轮改动边界
  -> 修改代码
  -> 按验证级别决定是否补测试
  -> 按验证级别决定是否跑测试
  -> 输出质量门禁结果
  -> 输出回滚方式
```

不要让 AI 一口气做多个阶段。

## 7. 对 AI 的架构提醒

关键句：

> 多项目不是核心 Orchestrator 的职责。核心 Orchestrator 仍然只管一个 workflow / 一个项目；Control Plane 负责启动和聚合多个 worker。

关键句：

> Web 不是 raw log viewer。Web 默认读后端 summary；timeline 和 raw payload 必须懒加载。

关键句：

> `WORKFLOW.md` 是项目契约。模块化只改变人维护的源文件结构，不改变 worker 读取最终 generated workflow 的方式。

